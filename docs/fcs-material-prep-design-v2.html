1. 业务背景与设计目标

1.1 背景

印尼供应链团队分出一个专门的配料小组，统一负责所有物料的配料操作。配料小组按照生产单 BOM，从各个仓库中挑出物料、拣货、放入暂存区、确认配料完成，然后通知下游来领料或配送。

核心业务规则：先配料，再分配任务。配料阶段只跟 BOM 走，不绑定工厂；任务分配时才确定具体工厂并触发配送。

1.2 系统归属

配料管理归属 FCS（工厂生产协同系统），作为独立一级菜单「配料管理」，包含 6 个子页面。

3. 核心数据模型

3.1 四层数据层次

配料单 (MaterialPrepOrder)          — 按生产单组织，一个生产单一张配料单
  └── 物料行 (MaterialPrepLine)      — 每个物料一行，含需求/库存/约束/上游进度
       └── 配料记录 (MaterialPrepRecord) — 多批次配料，每条独立生命周期
            ├── 配料记录明细 (MaterialPrepRecordItem)  — 记录内物料明细
            ├── 领料记录 (PickupRecord)                — 下游领料后的记录
            ├── 打回记录 (PrepRejectRecord)             — 下游打回中转仓的记录
            └── 暂存记录 (MaterialPrepStagingRecord)    — 入暂存区的追溯记录

4. 五大配料类型与物料路由

4.1 分类规则

每条物料行由 classifyPrepLineType()（#L708）根据 materialType + upstreamSourceType 自动路由：

配料类型路由条件taskLinks工厂配货仓库下一步
染色配料面料 + upstreamSourceType=染色已分配染色厂待加工仓加工领料
印花配料面料 + upstreamSourceType=印花已分配印花厂待加工仓加工领料
裁片配料面料 + 无印染上游已分配中转仓（印染成品面料）裁片工厂领料
车缝配料辅料 / 纱线未分配辅料仓 / 纱线仓配送（分配决定）
其他配料包材未分配包材仓配送（分配决定）
工艺绑定 vs 分配决定： 染色/印花/裁片任务是工艺绑定的（只能由特定类型工厂执行），因此 buildTaskLink 会通过 resolveFactoryForTask 从工厂档案解析具体工厂，标记为"已分配"。 车缝/包装任务不绑定具体工厂类型（多个车缝厂可承接），因此保持"未分配"，等任务分配后由 writeBackTaskAssignment 回写。

派生逻辑

由 deriveOrderPrepStatus（#L2059 附近）从配料记录状态和物料行完成度派生：

  • 任何 REJECTED 记录 → REJECTED_REWORK
  • 所有行 confirmedPrepQty ≥ requiredQty → READY
  • 所有行 availableStockQty ≥ remainingNeedQty → ALL_STOCK
  • 部分行有库存 → PARTIAL_STOCK
  • 全部无库存 → NO_STOCK

8. 配料数量计算规则

8.1 核心公式

染色 / 印花 / 裁片配料

maxPrepQty = min(remainingNeedQty, availableStockQty)

defaultPrepQty = maxPrepQty

车缝配料

maxPrepQty = min(remainingNeedQty, round(releaseQty × qtyRatio), availableStockQty)

defaultPrepQty = maxPrepQty

受裁片放行数量 + 库存数量双重约束

其他配料

maxPrepQty = min(remainingNeedQty, availableStockQty)

无放行约束

以上逻辑在 buildLine 函数（约 #L2049）中实现。外层有 Math.max(0, ...) 保护。

9. 车缝配料与裁片放行约束

车缝配料（辅料配料）的 maxPrepQty 受裁片放行数量约束。数据链路：

裁床主管判断 → cut-piece-release.ts
                └── getCutPieceReleaseSummaryForProductionOrder(productionOrderId)
                    返回 CutPieceReleaseSummary { decision, releaseQty, reason, ... }

sewing.ts → renderCutPieceReleaseCard() → 展示约束卡片
         → renderOrderMaterialRows() → 展示 maxPrepQty / defaultPrepQty 列

Fallback 机制

如果生产单尚未进入裁片放行管理（无放行记录），getCutPieceReleaseSummaryForProductionOrder 会返回一个默认的 fallback 记录（decision="可以做"，releaseQty=1000），使车缝配料页面不至于无数据可展示。

11. 暂存区管理

11.1 暂存区概念

仓库内的指定物理区域。配料完成后货放在暂存区等待下游来取。每个配料类型的暂存区位置不同：

配料类型配货仓库暂存区通知对象
染色配料染色厂待加工仓染色厂暂存区染色厂（加工领料）
印花配料印花厂待加工仓印花厂暂存区印花厂（加工领料）
裁片配料中转仓中转仓暂存区裁片工厂
车缝配料辅料仓/纱线仓相应暂存区分配后确定
其他配料包材仓包材仓暂存区分配后确定

11.2 暂存区数据模型

MaterialPrepStagingRecord（#L193）：包含 stagingRecordId、prepRecordId、stagingArea、stagedAt、stagedBy、itemCount、totalPreparedQty、warehouseNames、status（暂存中/已确认/已打回）。

每次 stageMaterialPrepRecord 都会新增一条 stagingRecord。confirmMaterialPrepRecord 和 rejectMaterialPrepRecord 会更新对应 stagingRecord 的状态。
