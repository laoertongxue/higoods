# HiGood 印尼工厂现场协同系统产品设计规范

## 1. 文档目的

本文档用于指导 HiGood 在印尼工厂场景下的产品设计和原型设计，覆盖工厂生产协同系统、工艺工厂运营系统、仓储物流系统、商品中心系统中会被工厂人员使用的页面。

这不是单纯的 UI 样式规范。它同时约束角色、流程、协作关系、任务表达、界面用语、视觉样式、组件交互、防错机制和原型数据。UI 样式是其中一层，但不能脱离业务协作和现场能力单独设计。

本规范的核心目标是：让低识字、低算术、低系统理解能力的一线员工，仍然能稳定参与复杂生产协作。

## 2. 适用范围

适用系统：

- FCS：工厂生产协同系统。
- PFOS：工艺工厂运营系统，包括印花、染色、裁床、特殊工艺、后道、毛织等工艺工厂 Web 页面。
- WLS：仓储物流系统，包括收货、领料、入仓、出仓、盘点、交接、差异。
- PCS：商品中心系统中会被工厂查看或执行的技术包、BOM、尺码、工艺、样衣、图片、物料资料。

适用端类型：

- 管理端：面向中国管理人员、计划、跟单、财务、运营、平台审核人员。
- 主管端：面向印尼工厂主管、组长、仓库负责人、质检负责人。
- 员工执行端：面向印尼本土一线员工，包括 PDA、工厂电脑、扫码枪、平板上的执行页面。

设计时按角色和任务分层，不按 Web / PDA 简单分层。Web 页面也可能是一线执行端，PDA 页面也可能存在主管模式。

## 3. 现场能力假设

印尼本土一线员工的现场能力必须作为产品设计硬约束，而不是培训问题。

默认能力假设：

- 识字能力不稳定：不能依赖长文案、抽象说明、复杂字段名。
- 数字能力不稳定：不能让员工心算数量、差异、剩余、比例、批次数。
- 流程理解弱：不能要求员工理解完整状态机或上下游系统模型。
- 抽象概念理解弱：不能把「执行对象」「来源记录」「投影」「写回」「链路」直接暴露给一线员工。
- 选择判断风险高：不能让员工在多个相似对象中靠文字判断。
- 纠错能力弱：错误发生后必须给出明确修正动作，必要时交给主管。
- 现场环境复杂：弱网、低分辨率、扫码枪、多人共用设备、噪音、赶工、跨语言都要纳入设计。

因此，员工执行端必须遵守：

- 少读。
- 少想。
- 少算。
- 少选。
- 少填。
- 系统多判断。
- 主管可兜底。

## 4. 体验设计三层

HiGood 的现场协同系统必须同时满足 3 个层次。

### 4.1 功能物层

功能物层关注按钮、表格、字段、弹窗、状态、颜色、图标、布局等界面对象。

这一层解决「能不能用」。例如：

- 是否有确认按钮。
- 是否有扫码入口。
- 是否展示数量。
- 是否有状态徽章。

仅满足这一层不够。一个页面有按钮、有表格、有弹窗，不代表现场能运转。

### 4.2 现场任务层

现场任务层关注一个人在具体场景里是否能完成任务。

这一层解决「用起来是否顺」。每个现场页面必须回答：

- 我是谁？
- 我现在处理哪个对象？
- 我现在只需要做什么？
- 做完后系统给我什么结果？
- 出错时我找谁？

员工执行端页面必须围绕「当前任务、当前对象、当前动作、当前结果」组织。

### 4.3 协作关系层

协作关系层关注系统是否支撑人与人之间的责任交接、信任建立和事实追溯。

这一层解决「系统能不能真正运转」。每个跨角色页面必须回答：

- 谁把什么交给谁？
- 谁确认了什么？
- 谁对数量、质量、时间负责？
- 差异发生后谁处理？
- 事实如何被 Web、PDA、仓储、结算共享？

例如，交接页面不是「展示交接字段」，而是让交出方、接收方、主管对数量和责任达成一致。

## 5. 角色与端类型

### 5.1 中国管理人员

目标：看全链路、查异常、做决策、追责任。

设计要求：

- 可以展示较高信息密度。
- 可以使用表格、筛选、统计、日志、抽屉。
- 必须能追溯事实来源。
- 必须能看到 Web / PDA / 仓储 / 结算之间的一致口径。

### 5.2 印尼主管

目标：分派任务、处理异常、复核结果、帮助员工兜底。

设计要求：

- 信息密度中等。
- 页面要突出待处理、异常、超时、差异。
- 允许查看更多上下文，但不能像管理端一样堆满全链路。
- 主管动作必须明确：分派、复核、确认差异、代处理、退回员工。

### 5.3 印尼一线员工

目标：完成现场动作。

设计要求：

- 低信息密度。
- 一页一个主动作。
- 扫码优先。
- 图片、颜色、尺码、袋号、库位等强识别优先。
- 不展示管理统计、复杂流水、系统状态机。
- 错误要阻断并给出下一步。

### 5.4 仓管

目标：证明物料、裁片、半成品、成衣在哪、由谁保管、数量是否一致。

设计要求：

- 收、发、退、调、盘点要分清。
- 所有数量必须带单位。
- 差异必须直接显示多 / 少多少。
- 库区、货架、库位要可扫码或快速选择。

### 5.5 质检

目标：记录质量事实，并把返工、扣款、放行、复检关系传递给后续角色。

设计要求：

- 缺陷、数量、照片、处理结论必须结构化。
- 一线质检端不应暴露复杂结算逻辑。
- 质量事实必须能被后续工序、结算、异常页追溯。

### 5.6 财务与结算人员

目标：核对金额来源、扣款依据、付款进度和工厂反馈。

设计要求：

- 必须保留操作人、时间、代确认原因、异议入口。
- 结算端可以复杂，但工厂查看端必须清楚说明「谁确认」「是否代确认」「还能不能异议」。

## 6. 系统分层原则

当前项目已经存在跨端统一口径。例如 `docs/fcs-process-work-order-unification.md` 明确：工厂生产协同系统展示平台视角，工艺工厂运营系统展示工厂 Web 视角，工厂端移动应用作为一线执行入口，三者读取同一个加工单 ID、加工单号、状态、任务引用和交出记录。

设计时必须遵守以下分层。

### 6.1 FCS：平台生产协同视角

FCS 关注全链路协同：

- 生产需求。
- 生产单。
- 工艺路线与任务拆解。
- 工艺加工单。
- 执行进度。
- 交出回写。
- 质检、异常、结算。

FCS 可以展示全局状态，但必须能跳转到现场事实，而不是只展示汇总结果。

### 6.2 PFOS：工艺工厂运营视角

PFOS 关注工艺工厂自身可执行的工单、仓、交接和异常。

工艺工厂 Web 不是单纯管理后台。若页面给印尼员工使用，应提供现场模式：

- 大按钮。
- 少筛选。
- 少表格。
- 扫码入口靠前。
- 当前动作明确。

### 6.3 WLS：仓储物流事实视角

WLS 关注物理对象的收发存和差异。

核心问题不是「库存表长什么样」，而是：

- 物在哪里？
- 谁收了？
- 谁发了？
- 数量差多少？
- 下一步谁处理？

仓储页面必须避免让一线员工在大量库存行中靠文字判断。

### 6.4 PCS：商品资料转执行视角

PCS 面向工厂时，不应展示商品中心内部复杂项目管理。

给工厂看的 PCS 信息必须转成现场可执行信息：

- 款式图片。
- 颜色。
- 尺码。
- 工艺要求。
- BOM 物料。
- 注意事项。
- 作业对象。

技术包不是资料归档，而是防止工厂做错的作业依据。

## 7. 任务与动作设计规范

### 7.1 一线端只表达当前任务

员工执行端必须优先展示：

- 当前任务。
- 当前对象。
- 当前动作。
- 当前数量。
- 当前结果。

禁止把统计、日志、长流水、全链路状态塞到员工执行页首屏。

### 7.2 一个页面一个主动作

员工执行端首屏只能有一个主动作。

允许存在辅助动作，例如返回、查看详情、叫主管，但不能和主动作同等视觉权重。

错误示例：

- 同一张卡片同时出现「查看」「加工领料」「回收入仓」「调整位置」。

推荐示例：

- 主按钮：「扫码领料」。
- 次按钮：「查看详情」「叫主管」。

### 7.3 系统判断下一步

员工不应该判断状态机。

系统应根据任务、扫码对象、数量和角色判断下一步。例如：

- 当前是待领料，则进入收货确认。
- 当前是加工中，则进入完工上报。
- 当前是待交出，则进入交出扫码。
- 当前有差异，则进入主管处理。

### 7.4 复杂动作拆步骤

复杂动作必须拆成短步骤：

1. 扫码。
2. 核对对象。
3. 输入或确认数量。
4. 拍照。
5. 确认。

每一步只问一个问题。

## 8. 信息架构与导航规范

### 8.1 不让员工找菜单

员工执行端不应让用户先理解「接单 / 执行 / 交接 / 仓管 / 结算」这种模块关系。

员工进入系统后，应优先看到：

- 我的待办。
- 当前要做的任务。
- 扫码入口。
- 异常提醒。

模块导航可以保留给主管端和管理端。

### 8.2 管理端可全局，执行端必须收敛

管理端允许：

- 多级菜单。
- 大表格。
- 高级筛选。
- 多 Tab。
- 抽屉详情。

员工执行端应避免：

- 多级菜单。
- 多 Tab 并列判断。
- 复杂筛选。
- 长表格。
- 横向滚动。

### 8.3 工作台优先

主管端和员工执行端应优先从工作台进入任务，而不是从功能菜单进入。

工作台必须突出：

- 今日待做。
- 超时。
- 差异。
- 待主管处理。
- 最近扫码结果。

## 9. 页面模式规范

### 9.1 工作台

管理端工作台看全局，主管端工作台看待处理，员工端工作台看当前动作。

员工端工作台首屏必须回答：

- 现在做哪一单？
- 扫哪里？
- 做完点什么？

### 9.2 列表页

管理端列表可以使用表格。

员工执行端列表应优先使用卡片，并限制字段：

- 任务号。
- 图片或颜色。
- 款号。
- 数量。
- 状态动作。
- 主按钮。

### 9.3 详情页

详情页要按角色分层。

管理端可以展示完整信息。员工端详情只展示防错所需信息：

- 图片。
- 颜色。
- 尺码。
- 袋号。
- 数量。
- 位置。
- 当前动作。

### 9.4 执行页

执行页必须像作业指导书，而不是像后台表单。

推荐结构：

1. 当前对象。
2. 系统识别结果。
3. 本步动作。
4. 必要输入。
5. 确认按钮。
6. 异常 / 叫主管。

### 9.5 仓储页

仓储页必须围绕「物在哪里」设计。

员工端优先：

- 扫单。
- 扫袋。
- 扫库位。
- 确认数量。
- 拍照。

管理端可展示库存、流水、库位、筛选和统计。

### 9.6 交接页

交接页必须表达责任转移。

必须清楚展示：

- 交出方。
- 接收方。
- 交出对象。
- 应交数量。
- 实交数量。
- 差异。
- 照片或扫码证据。

### 9.7 质检页

质检页必须表达质量事实。

员工端要简化成：

- 合格数量。
- 不合格数量。
- 缺陷类型。
- 照片。
- 处理动作。

结算影响可以后置到管理端。

### 9.8 技术资料页

给工厂看的技术资料页必须突出现场识别：

- 款式图。
- 颜色图。
- 尺码。
- 工艺图片。
- 物料图片。
- 注意事项。

不要只给复杂表格和内部编码。

## 10. 界面用语与现场表达规范

界面用语借鉴华为界面用语原则：清晰、简洁、准确、一致、友好、尊重。HiGood 现场版要进一步强调动作化和防错。

### 10.1 动作优先

按钮和提示优先写用户要做的动作。

推荐：

- 「扫码领料」
- 「确认已交出 20 件」
- 「拍照上传」
- 「叫主管处理」

避免：

- 「提交」
- 「处理」
- 「执行流程」
- 「确认状态变更」

### 10.2 短句优先

一句话只表达一个意思。

不要在同一句里同时写原因、规则、状态和后果。

### 10.3 用户词优先

员工执行端禁止直接暴露系统抽象词：

- 写回。
- 投影。
- 执行对象。
- 来源记录。
- 状态流转。
- 链路。

应替换为现场动作：

- 扫袋子。
- 收货。
- 交出。
- 拍照。
- 叫主管。

### 10.4 错误必须给改法

禁止只写：

- 「操作失败」
- 「输入错误」
- 「参数异常」

必须写：

- 「最多可交出 20 件，请改成 20 件以内。」
- 「这个袋子不属于当前任务，请扫描 TASK-001 的袋子。」
- 「照片上传失败，请重新拍照；仍失败请叫主管。」

### 10.5 同一动作全系统同名

同一动作不能在不同系统里混用多个叫法。

需要统一的动作词包括：

- 接单。
- 领料。
- 收货。
- 入仓。
- 开工。
- 完工。
- 交出。
- 质检。
- 复检。
- 上报差异。
- 叫主管。

### 10.6 尊重但不含糊

不要责备用户。

推荐：

- 「这个袋子不属于当前任务。」

避免：

- 「你扫错了。」

## 11. 数据、数量与状态表达规范

### 11.1 不让员工心算

所有数量必须由系统计算：

- 已领。
- 应领。
- 剩余。
- 已交。
- 应交。
- 差异。

员工最多输入实际看到的数量。

### 11.2 数量必须带单位

数量必须带单位，并且单位要符合现场对象。

示例：

- `20 件`
- `120 片`
- `3 包`
- `15 米`

不要只显示数字。

### 11.3 差异要直接表达

推荐：

- 「少 2 件」
- 「多 1 包」
- 「数量一致」

避免：

- `diffQty = -2`
- 「存在差异」

### 11.4 状态要翻译成动作

后台可以使用复杂状态，员工端要翻译成动作。

示例：

| 后台状态 | 员工端表达 |
| --- | --- |
| 待交出 | 现在交出 |
| 交出待收货 | 等对方收货 |
| 收货差异 | 数量不一致，叫主管 |
| 待回写 | 等系统同步 |

## 12. 扫码、图片与识别规范

### 12.1 扫码优先

能扫码就不要让员工手选。

优先扫码对象：

- 任务单。
- 交接单。
- 袋子。
- 菲票。
- 库位。
- 物料。
- 成衣箱。

### 12.2 图片优先

现场识别对象时，应优先使用图片辅助：

- 款式图。
- 颜色图。
- 裁片图。
- 物料图。
- 包装照片。
- 标签照片。

### 12.3 文字编号不能单独承担识别

编号可以保留，但不能只靠编号识别。

编号必须配合：

- 图片。
- 颜色。
- 尺码。
- 数量。
- 位置。

## 13. 防错与异常兜底规范

### 13.1 防错优先于提醒

系统能阻断的错误，不要只提醒。

必须阻断：

- 超量。
- 扫错对象。
- 重复提交。
- 必填缺失。
- 状态不允许操作。

### 13.2 主管兜底必须可见

员工端遇到无法判断的问题，必须有明确入口：

- 「叫主管处理」
- 「上报差异」
- 「拍照说明」

### 13.3 弱网和上传失败要可恢复

上传和提交失败时，页面必须告诉用户：

- 当前是否保存。
- 是否需要重试。
- 是否可以继续拍照。
- 是否需要主管处理。

### 13.4 危险动作要二次确认

涉及删除、作废、差异确认、代确认、结算推进、库存调整的动作，必须有二次确认和操作记录。

## 14. UI 样式与视觉规范

### 14.1 整体风格

整体风格应保持企业后台的清晰、稳定、克制，但现场端必须更像操作台。

要求：

- 管理端可密集。
- 主管端中等密度。
- 员工执行端低密度。
- 异常突出。
- 装饰克制。

### 14.2 字号与层级

员工执行端：

- 主标题要明显。
- 主按钮文字要足够大。
- 关键数量要大号展示。
- 次要说明要短。

管理端：

- 可使用较密表格。
- 但关键异常和操作入口仍要突出。

### 14.3 颜色

颜色必须服务识别，不做装饰。

要求：

- 红色只用于错误、危险、阻断。
- 黄色用于待处理、临近风险。
- 绿色用于已完成、数量一致。
- 蓝色用于主动作或信息提示。
- 禁止满屏红黄绿导致注意力失焦。

### 14.4 图标

图标用于辅助理解，不替代关键文字。

员工执行端不得只显示图标按钮，必须有文字说明。

### 14.5 按钮

员工执行端：

- 主按钮必须唯一且明显。
- 按钮可点击区域要大。
- 危险按钮必须远离主按钮。

管理端：

- 行操作可以紧凑，但危险动作仍需二次确认。

### 14.6 表格

管理端可以使用表格。

员工执行端慎用表格，优先卡片、步骤、扫码结果。若必须使用表格：

- 列数要少。
- 不横向滚动。
- 操作按钮清晰。

### 14.7 卡片

卡片用于任务、物料、交接对象、异常对象。

不要为了装饰堆叠卡片。卡片必须承载可执行信息。

### 14.8 图片

现场端图片优先级高于文字描述。

图片必须清晰展示真实对象，不用模糊、裁切过重、无法识别的图。

### 14.9 低分辨率

必须考虑：

- PDA 小屏。
- 工厂电脑低分辨率。
- 1024 x 768。
- 扫码枪输入。

低分辨率下不得出现主要按钮不可见、表单横向溢出、弹窗无法提交。

## 15. 组件与交互规范

### 15.1 筛选

员工执行端尽量不使用复杂筛选。

若需要筛选，优先使用：

- 今日。
- 待我处理。
- 异常。
- 已完成。

### 15.2 Tabs

Tabs 适合管理端和主管端。

员工执行端仅在状态非常清楚时使用，且数量不宜超过 3 个。

### 15.3 弹窗与抽屉

管理端可使用抽屉展示详情。

员工执行端弹窗必须短：

- 不放长表单。
- 不放复杂说明。
- 主要按钮可见。
- 低分辨率下可提交。

### 15.4 表单

员工执行端表单字段要少。

优先：

- 扫码带出。
- 默认值。
- 步进器。
- 大按钮选择。

避免大量手输。

### 15.5 上传

上传必须说明：

- 拍什么。
- 是否成功。
- 失败怎么办。
- 是否可重拍。

### 15.6 状态徽章

状态徽章必须可理解。

员工端状态徽章应尽量转成动作词，不展示英文码或抽象内部状态。

### 15.7 步骤条

步骤条用于让用户知道当前在哪一步。

员工端步骤数量宜少，建议 3 到 5 步。

## 16. 原型数据与演示要求

原型数据不能只有空壳。

必须体现：

- 上游来源。
- 下游去向。
- 多角色协作。
- 多状态。
- 差异。
- 异常。
- 照片或扫码证据。
- 可追溯操作记录。

Mock 数据应优先使用中文业务语义，不展示英文状态码。若涉及印尼现场端，应考虑后续可翻译为印尼语的短句表达。

原型演示时，必须能说明：

- 这个页面服务哪个角色。
- 这个页面承担哪段协作关系。
- 这个页面如何防止现场员工做错。
- 这个页面的事实如何传递到上游或下游。
