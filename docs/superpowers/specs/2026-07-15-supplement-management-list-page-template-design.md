# 补料管理标准列表页模板设计规格

## 1. 背景与目标

当前仓库已经提供 `button`、`filter-bar`、`table`、`pagination`、`card` 等字符串模板 UI 组件，但列表页对标题、筛选、统计、表格、分页和宽表能力的组合方式不统一，部分页面仍存在自行拼装分页、装饰性说明文案和低分辨率下操作入口不可见的问题。

本次只调整工艺工厂运营系统中的“补料管理”页面，将它建设为仓库内第一个可运行的标准列表页模板，并把模板要求写入项目根目录 `AGENTS.md`。后续新增列表页或调整既有列表页时，Agent 必须参照该模板并复用公共组件。

标准模板页面与入口：

- 页面名称：补料管理
- 页面路由：`/fcs/craft/cutting/supplement-management`
- 页面文件：`src/pages/process-factory/cutting/supplement-management.ts`

## 2. 范围

### 2.1 本次范围

- 新增轻量公共列表页骨架组件。
- 必要时完善现有筛选、表格和分页基础组件。
- 让补料管理列表页完整接入公共骨架和基础组件。
- 为补料管理增加标准分页、宽表列设置和固定操作列能力。
- 将标准列表页规则写入项目根目录 `AGENTS.md`。
- 完成设计治理记录、自动检查和低分辨率浏览器验收。

### 2.2 不在本次范围

- 不批量迁移或调整其他现有列表页。
- 不改变补料管理的新增补料流程、详情业务规则或补料数据模型。
- 不建立配置驱动的通用页面生成器。
- 不引入 React、状态管理框架或新的 UI 框架。
- 不强制 PDA、卡片工作台和非列表页面使用本模板。
- 不实现账号级或服务端的列偏好同步。

## 3. 设计原则

### 3.1 模板是可运行页面，不是静态示意

补料管理必须真实使用公共列表页骨架、表格、筛选和分页组件。后续 Agent 既可以从 `AGENTS.md` 获取约束，也可以从补料管理查看实际组合方式，不能依赖复制一段独立页面模板。

### 3.2 统一结构，不接管业务

公共骨架只负责区域顺序、样式、响应式布局和通用交互协议。页面继续负责业务数据、筛选条件、统计口径、详情弹窗、新增流程和业务校验，避免形成复杂的配置引擎。

### 3.3 禁止无业务逻辑的说明性文案

列表页任一模块都不允许放置固定的解释、用途说明或装饰性提示。禁止示例包括：

- “本页面用于……”
- “列表对象是……”
- “新增后将……”
- 对页面结构、操作流程或字段含义的重复解释

允许保留会随业务状态变化且能指导操作的内容：

- 业务状态反馈
- 校验错误
- 风险警告
- 空状态提示
- 字段标签和输入占位提示

## 4. 标准页面结构

列表页区域顺序固定为：

1. 页面标题与主要业务操作
2. 筛选区
3. 紧凑统计区，可选
4. 数据列表区，空数据时在表格内展示空状态
5. 标准分页区

### 4.1 标题区

- 只显示页面名称和主要业务操作。
- 补料管理保留“补料管理”和“新增补料”。
- 删除页面标题下方的面包屑路径。
- 不显示副标题、用途说明或操作说明。

### 4.2 筛选区

- 筛选区保留统一边框，用于明确输入范围。
- 优先单行排列；宽度不足时按筛选字段、查询操作、重置操作的顺序换行。
- 关键词输入不得在每次 `input` 时触发整页重绘，应采用提交筛选、防抖或局部更新。
- 筛选和重置后回到第 1 页。

### 4.3 统计区

- 统计区只显示指标名称和数值。
- 各指标使用独立紧凑信息块。
- 指标组不设置整体外框、背景、圆角容器或阴影。
- 补料管理保留“补料单”“已确认”“涉及生产单”。
- 统计口径基于筛选后的完整结果，不按当前页数据计算。

### 4.4 列表区

- 列表容器使用统一边框、表头、行高、空状态和分页结构。
- 列表头只显示业务对象名称和必要业务操作。
- 删除补料管理列表头右侧的“列表对象是补料单；新增补料填写后会弹窗确认。”
- 核心识别列优先可见，次要信息可以折行、截断或进入详情。
- 操作列始终固定在最右侧，不受横向滚动影响。

## 5. 公共组件架构

新增轻量公共列表页骨架，建议放在 `src/components/ui/list-page.ts`。骨架采用 HTML 插槽组合，不读取或修改业务数据。

骨架提供以下受约束参数：

- `title`：页面名称。
- `primaryActionsHtml`：新增等主要业务操作。
- `filtersHtml`：筛选字段及查询、重置。
- `statsHtml`：可选的紧凑统计项。
- `listTitle`：列表业务对象名称。
- `tableHtml`：业务表格。
- `paginationHtml`：标准分页。

骨架接口不提供 `description`、`subtitle`、`hint` 等静态说明字段，从组件层面减少说明性文案进入列表页的可能。

现有基础组件的职责保持清晰：

- `button.ts`：主要和次要操作。
- `filter-bar.ts`：搜索、筛选、查询和重置。
- `table.ts`：表头、数据行、空状态、冻结列和列设置需要的展示能力。
- `pagination.ts`：统一分页栏和分页事件协议。
- `drawer.ts` 或 `dropdown.ts`：列设置入口与设置面板。

## 6. 分页规则

所有数据列表都必须展示标准分页，即使当前只有一页或没有数据。

- 默认每页 10 条。
- 可选每页条数为 10、20、50 条。
- 展示总条数、当前数据范围、当前页、总页数、上一页和下一页。
- 筛选、重置或修改每页条数后回到第 1 页。
- 数据变化导致当前页越界时，自动回到最后一个有效页。
- 排序作用于筛选后的全部数据，排序完成后再分页。
- 表格只渲染当前页数据，避免一次性将大量不可见行写入 DOM。
- 翻页优先局部更新表格和分页区域，不触发整页闪烁或滚动位置丢失。

## 7. 宽表与列管理

字段较多的列表页在列表标题区域提供“列设置”入口。普通字段较少的列表页可以不展示该入口，但仍需使用统一表格结构并固定右侧操作列。

### 7.1 列显示与顺序

- 用户可以显示或隐藏可选列。
- 用户可以拖动调整列的左右排列顺序。
- 核心识别列可被标记为不可隐藏。
- 操作列不可隐藏、不可移动，始终位于最右侧。
- 提供“恢复默认设置”。

### 7.2 冻结列

- 用户可以将业务列冻结在表格左侧。
- 左侧冻结列按当前列顺序排列。
- 不提供将普通业务列冻结到右侧的能力。
- 操作列由系统强制冻结在右侧。
- 系统根据可视宽度限制左侧冻结列的总宽度；达到安全上限时阻止继续冻结并给出明确反馈。
- 中间非冻结区域允许在列表容器内部横向滚动，页面本身不得横向溢出。

### 7.3 数据排序

- 支持排序的字段在表头显示排序状态。
- 点击表头依次切换升序、降序和默认顺序。
- 首版使用单字段排序。
- 排序作用于筛选后的完整数据，再执行分页。
- 隐藏当前排序列时，自动取消该字段排序。
- 冻结、显隐和列顺序只改变展示，不修改业务数据。

## 8. 列设置保存规则

列偏好按页面路由分别保存，原型阶段使用浏览器本地存储。

长期保存：

- 列显隐
- 列顺序
- 左侧冻结列
- 每页条数

不长期保存：

- 当前页码
- 当前数据排序

重新进入页面时回到第 1 页并使用默认排序。“恢复默认设置”清除当前页面的本地偏好并恢复系统默认列结构。

若本地配置损坏，页面自动回退到系统默认值。若浏览器禁止本地存储，列设置在当前页面会话内仍然可用，不得阻断列表业务。

## 9. 低分辨率适配

主要设计与验收尺寸为 `1366×768`，最低可用尺寸为 `1280×720`。

- 页面外边距和模块间距采用紧凑规格，区域间距控制在 12–16px。
- 标题、筛选、统计和列表不得使用无业务价值的大面积留白。
- 筛选项可有序换行，查询和重置按钮不能被挤出可视区域。
- 统计区使用紧凑横排，不使用大 KPI 卡片。
- 表头、单元格、图片和行高采用紧凑规格。
- 核心识别列和右侧操作列优先可见。
- 宽表在容器内部滚动，右侧操作列始终可见。
- 分页紧贴表格底部，在最低尺寸下仍可完成每页条数选择和翻页。
- 弹窗与抽屉限制最大视口高度，内容区滚动，底部操作区保持可见。

在 `1280×720` 下，用户必须能够完成筛选、查看列表、翻页、调整列和使用行操作。

## 10. Agent 治理规则

项目根目录 `AGENTS.md` 新增“标准列表页模板”章节，至少包含以下强制规则：

- 补料管理是标准列表页模板，并列出路由和代码文件。
- 新增列表页必须使用公共列表页骨架及已有基础组件。
- 调整既有列表页的页面结构、筛选、统计、表格或分页时，必须向模板规范对齐。
- 当前任务不要求批量迁移其他已有列表页。
- 所有数据列表必须分页。
- 所有操作列必须固定在右侧。
- 宽表必须支持列显隐、列顺序、左侧冻结和数据排序。
- 列表页任一模块禁止没有实际业务逻辑的说明性文案。
- 使用 `1366×768` 做主要验收，并保证 `1280×720` 可完成核心操作。
- 若业务确实不能使用标准结构，必须在对应原型审查记录中说明原因和例外范围，不能自行复制另一套列表组件。

## 11. 异常与空状态

- 没有数据时展示与当前业务对象相关的空状态，并保留分页数据口径。
- 不支持排序的字段不显示排序入口。
- 列配置损坏或字段版本变化时忽略无效字段并恢复可用默认值。
- 固定列总宽度超过安全阈值时阻止操作，并明确说明需要先取消其他冻结列。
- 浏览器本地存储异常不得造成白屏或阻断筛选、分页和查看详情。
- 所有运行时反馈必须是中文业务文案，不显示内部状态码。

## 12. 验证与验收

### 12.1 功能验证

- 补料管理路由可达，现有新增补料和查看详情业务保持可用。
- 筛选、重置、分页和每页条数行为符合统一规则。
- 统计指标基于筛选后的完整数据。
- 列显隐、列顺序、左侧冻结和恢复默认设置有效。
- 表头升序、降序、默认顺序切换有效，排序后分页结果正确。
- 操作列在横向滚动时始终固定在右侧。
- 本地列偏好刷新后保留，损坏配置能自动回退。

### 12.2 视觉与交互验证

- 在 `1366×768` 和 `1280×720` 两个视口下进行浏览器验证。
- 页面不存在面包屑、副标题或无业务逻辑说明文案。
- 筛选项、主要按钮和分页操作在低分辨率下可见、可用。
- 横向滚动只发生在表格容器内。
- 筛选、翻页、列设置、弹窗和行操作不发生不必要的整页重绘。
- 任一功能按钮的可感知响应时间不超过 200ms。

### 12.3 工程与治理验证

- 运行与模板组件和补料管理相关的最小检查。
- 运行 `npm run build`。
- 运行 `npm run check:prototype-design-governance`。
- 新增或更新 `docs/prototype-review-records/` 下的补料管理审查记录。
- 修改完成后同步 CodeGraph 并确认索引状态正常。

## 13. 完成标准

满足以下条件才视为本次模板建设完成：

1. 补料管理使用公共列表页骨架和统一基础组件。
2. 页面符合无说明性文案、强制分页、低分辨率和固定操作列要求。
3. 宽表列显隐、列顺序、左侧冻结、数据排序和本地保存均可演示。
4. `AGENTS.md` 已写入可执行的列表页强制规范，并明确引用补料管理模板。
5. 其他既有页面未被批量迁移或无关修改。
6. 构建、设计治理检查和两个低分辨率视口的浏览器验收通过。
