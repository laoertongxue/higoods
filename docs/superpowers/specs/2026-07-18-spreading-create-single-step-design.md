# 新建铺布单步选择设计

## 1. 背景

当前“新建铺布”采用两步流程：先按唛架方案选择，再进入确认页批量生成该方案下全部唛架编号对应的铺布单。

该流程与实际操作不一致：业务人员通常先用生产需求单、生产单、SPU 或裁片单定位业务，再从关联唛架方案中选择一个具体唛架编号创建一张铺布单。现有页面还缺少每个唛架编号的每层尺码件数、已有铺布单状态和款式图片，容易造成选错或重复创建。

## 2. 目标

- 将“新建铺布”从两步页面改为单页完成。
- 以上游业务对象作为主要搜索入口。
- 搜索结果按唛架方案分组，并展示方案下全部唛架编号。
- 每次只能选择一个尚未生成铺布单的唛架编号，并生成一张铺布单。
- 同时展示每个唛架编号的每层尺码件数和已有铺布单状态。
- 展示款式 / SPU 缩略图，并支持在当前页面查看大图。

## 3. 非目标

- 不改变唛架方案、唛架编号、铺布单之间的业务对象关系。
- 不改变铺布单生成模型、铺布执行流程和生成后进入编辑页的行为。
- 不新增后端、接口、状态管理体系或通用图片平台。
- 不重做铺布列表、详情和编辑页面。
- 不新增物料不足、捆条加工或铺布状态规则，仅沿用现有规则。

## 4. 角色与任务

| 项目 | 设计口径 |
| --- | --- |
| 系统 | PFOS 裁床模块 |
| 端类型 | 管理端 / 主管端 Web 页面 |
| 主要角色 | 裁床计划员、裁床主管 |
| 当前任务 | 从上游业务对象关联的唛架方案中，选择一个可用唛架编号并生成一张铺布单 |
| 上游来源 | 生产需求单、生产单、SPU、裁片单、唛架方案 |
| 下游去向 | 新生成的待铺布单编辑页 |

## 5. 核心业务规则

1. 一个唛架方案可以包含多个唛架编号。
2. 一个唛架编号最多对应一张有效铺布单。
3. 每次新建只能选择一个具体唛架编号，并生成一张铺布单。
4. 已生成铺布单的唛架编号必须继续展示，但不可再次选择。
5. 已生成行必须展示铺布单号和当前中文状态，并提供铺布单详情入口。
6. 每个唛架编号的尺码表读取该编号的“每层各尺码件数”，不得用计划总件数代替。
7. 搜索生产需求单、生产单、SPU 或裁片单时，可以返回多个唛架方案；页面按唛架方案分组展示。

## 6. 页面流程

页面不再显示步骤条，也不再跳转到第二步确认页。

1. 用户输入生产需求单号、生产单号、SPU 或裁片单号。
2. 系统识别搜索对象，并展示关联需求单、生产单、SPU 和裁片单摘要。
3. 系统列出该对象关联的全部唛架方案。
4. 用户在某个方案中选择一个尚未生成铺布单的唛架编号。
5. 底部操作区显示将要生成的方案和编号。
6. 用户点击“生成铺布单”。
7. 系统生成一张待铺布单，并进入该铺布单编辑页。

关联捆条加工单时，保留现有风险确认弹窗。该弹窗是生成动作的风险确认，不构成第二个页面步骤。

## 7. 信息架构

### 7.1 业务搜索区

主搜索框支持：

- 生产需求单号。
- 生产单号。
- SPU。
- 裁片单号。

系统自动识别输入对象。唛架方案、唛架编号和颜色放入“更多筛选”，作为辅助条件，不占据主要搜索入口。

搜索区只保留“查询生产任务”外层区块，不再嵌套带边框、圆角和阴影的筛选卡片。生产业务查询输入框、“查询”主按钮和“重置”次按钮直接单行排列；窄屏允许自然换行。

搜索成功后展示识别结果，例如：

> 已识别：生产单 PO-202603-0008；关联需求单 DEM-202603-0014、SPU-2024-014、2 张裁片单、3 个唛架方案。

### 7.2 结果摘要

结果摘要展示：

- 唛架方案数量。
- 唛架编号总数。
- 可新建数量。
- 已生成数量。

摘要数量由当前搜索结果自动计算，不要求用户心算。

### 7.3 唛架方案分组

每个方案头部展示：

- 唛架方案编号。
- 生产需求单、生产单和裁片单。
- 款式编码、SPU 和颜色。
- 款式 / SPU 缩略图。
- 方案内唛架编号总数和可新建数量。

仍有可新建编号的方案排在前面并默认展开。全部已生成的方案保留展示，但默认收起。

### 7.4 唛架编号明细

每个编号展示：

- 选择动作。
- 唛架编号。
- 尺码表（每层件数）。
- 计划层数。
- 计划总件数。
- 铺布单号和铺布单状态。

尺码表只展示件数大于 0 的尺码，统一格式为：

> M × 2 + L × 3 + XL × 1

尺码比例中不重复展示“件/层”，也不另列每层合计件数。

已有铺布单的行禁用选择按钮，并通过铺布单号进入详情。

### 7.5 底部操作区

未选择时，主按钮禁用。

选择后展示：

> 将为 MKP-20260403-007 / A-1 生成 1 张铺布单。

主按钮文案为“生成铺布单”。

## 8. 款式图片

- 每个唛架方案头部展示款式 / SPU 缩略图，方案内编号不重复展示图片。
- 点击缩略图后，在当前页面打开大图弹窗。
- 弹窗支持遮罩点击、关闭按钮和 Esc 键关闭。
- 缺少图片时展示“暂无款式图”占位，不影响唛架编号选择。
- 图片只承担款式识别作用，不改变商品资料或图片维护流程。

## 9. 交互与状态

### 9.1 搜索

- 支持回车和“查询”按钮。
- 输入过程不触发整页重绘。
- 输入内容先保留为待查询值，只有点击“查询”或按 Enter 后才更新结果。
- “重置”立即清空待查询值和已生效关键词，并恢复全部方案。
- 新搜索结果生效后清除旧选择。
- 识别到多个同名对象时，先展示可选对象，不直接混合其唛架方案。

### 9.2 选择

- 全页只能选择一个唛架编号。
- 选择新编号时自动取消旧编号。
- 选择只局部更新编号行和底部操作区。
- 已生成铺布单的编号不可选择。

### 9.3 生成

- 点击生成后立即禁用主按钮，防止重复点击。
- 提交前再次校验该编号是否已有有效铺布单。
- 校验失败时阻断生成，并刷新展示已有铺布单号和状态。
- 生成成功后进入新铺布单编辑页。
- 若关联捆条加工单，沿用现有二次确认内容和取消反馈。
- 捆条提示卡片右上角可关闭；提示内展示捆条名称、宽度和切割方式组成的捆条规格，不展示捆条菲票编号。

## 10. 空态与异常

| 场景 | 页面反馈 |
| --- | --- |
| 未输入关键词 | 提示输入生产需求单、生产单、SPU 或裁片单 |
| 未识别业务对象 | 明确展示未找到的输入内容，并允许重新搜索 |
| 识别对象但无唛架方案 | 提示当前业务对象尚未形成唛架方案 |
| 有方案但无可铺布编号 | 展示全部已有铺布单及状态，不展示可用主按钮 |
| 图片缺失 | 展示“暂无款式图”占位 |
| 重复生成 | 阻断创建，并展示已存在铺布单入口 |
| 捆条确认取消 | 保持当前选择并展示已取消反馈 |

## 11. 数据与投影口径

新建页的单行数据仍以具体唛架编号为粒度。结果展示层按唛架方案分组，不合并业务对象。

单个唛架编号需要提供：

- 唛架方案标识与编号。
- 唛架编号标识与编号。
- `sizePiecePerLayer` 对应的每层尺码件数。
- 计划层数和计划总件数。
- 款式编码、SPU、颜色和图片地址。
- 已有铺布单标识、单号和中文状态。
- 是否允许新建。

现有铺布单事实优先于页面缓存。页面缓存仅用于交互展示，不得作为重复创建判断的唯一依据。

## 12. 实现边界

采用“重写新建铺布子页面、保留外围链路”的方式：

- 移除新建页两步状态、步骤条和第二步确认页面。
- 保留现有新建路由。
- 保留现有铺布单生成模型、持久化入口和生成后导航。
- 保留现有捆条确认和物料状态规则。
- 将业务搜索、方案分组、编号行、图片弹窗和底部操作区拆成独立渲染函数。
- 使用现有字符串模板 UI 组件和局部 DOM 更新方式。
- 不重构铺布列表、详情、编辑和其他裁床模块。

## 13. 性能与页面适配

- 搜索输入不触发 `root.innerHTML` 级整页重绘。
- 选择编号、展开方案和打开图片只更新必要区域。
- 图标只在新增区域内初始化。
- 唛架方案结果分页，并明确方案总数、当前页和每页条数。
- 1366 × 768 为标准验收分辨率，1280 × 720 为最低可用分辨率。
- 页面主体不横向溢出；编号宽表只在结果容器内部滚动。
- 轻交互点击后应在 200 ms 内产生可见反馈。

## 14. 验收标准

1. 分别使用生产需求单、生产单、SPU 和裁片单完成搜索。
2. 搜索可以返回一个或多个唛架方案，并正确展示关联关系。
3. 同一方案可以同时展示未生成、待铺布、铺布中和已完成的编号。
4. 每个编号展示正确的每层尺码件数，且不混用计划总件数。
5. 全页只能选择一个未生成编号，并只生成一张铺布单。
6. 已生成编号不可选择，可进入已有铺布单详情。
7. 缩略图可以打开和关闭大图，缺图时显示占位。
8. 重复生成被阻断，捆条确认规则保持有效。
9. 空结果、无唛架方案和全部已生成场景均有明确反馈。
10. 搜索、选择、展开、图片弹窗和生成操作无明显整页闪烁。
11. 通过页面相关检查、设计治理检查和项目构建。
12. 在本地浏览器验证新建铺布路由，并覆盖 1366 × 768 与 1280 × 720。

## 15. 原型审查要求

实现时必须新增对应的原型审查记录，至少覆盖：

- 角色匹配和任务清晰度。
- 业务对象识别和方案分组是否容易理解。
- 尺码、层数和件数是否无需人工心算。
- 已生成编号是否有明确阻断和详情入口。
- 款式图片是否能辅助识别。
- 低分辨率和横向滚动是否可用。
- 是否存在列表页治理例外；如有，记录原因、影响和替代措施。

交付前运行 `npm run check:prototype-design-governance`，并按实际影响运行相关页面检查和 `npm run build`。
