# 配料详情摘要区清理设计

## 目标

配料管理下的各分类配料详情页统一参照裁片配料详情页处理：删除详情页 Tab 上方的大块摘要区，并把其中仍有业务判断价值的信息迁移到对应 Tab 内。

本次只处理配料详情页，不处理列表页顶部统计区，也不扩展仓储、质检、换料、补料等后续业务。

## 范围

涉及页面：

- `/fcs/material-prep/dyeing`
- `/fcs/material-prep/printing`
- `/fcs/material-prep/sewing`
- `/fcs/material-prep/other`
- `/fcs/material-prep/cutting`

说明：

- 裁片配料已完成主要形态，本次只检查并保持一致。
- 配料列表页 `/fcs/material-prep/list` 如果没有同类详情摘要区，不纳入改动。

## 当前问题

多个分类配料详情页在 Tab 上方同时展示两类摘要：

- 业务状态 KPI：配料状态、领料状态、物料行、缺料缺口。
- 实现链路摘要：BOM 来源、暂存区台账、仓库拣货进度、完成通知、分配回写。

这些信息放在 Tab 外会造成：

- 详情页首屏过重。
- 与 Tab 内内容重复。
- 用户进入详情后先看到一组孤立卡片，不知道应在哪个 Tab 继续判断或处理。

## 设计原则

- 删除 Tab 上方整块摘要区，不保留空容器。
- 不丢失有业务判断价值的信息。
- 信息迁移到现有 Tab，不新增新的主 Tab。
- 不做大模板重构；优先复用现有页面结构和本地函数。
- 不改变配料、领料、退回的业务状态和 mock 数据。

## 信息迁移规则

| 原摘要信息 | 迁移位置 | 处理方式 |
| --- | --- | --- |
| 配料状态 | `生产需求信息` 或 Tab 标题附近 | 作为当前配料单基础状态展示 |
| 领料状态 | `与配料记录关联的领料记录` | 与领料记录、可领/已领数量放在一起 |
| 物料行 | `当前各仓库存信息与上游进度` | 与库存、缺料、上游进度一起展示 |
| 缺料缺口 | `当前各仓库存信息与上游进度` | 保留缺口数量和最早可配时间 |
| BOM 来源 | `生产需求信息` | 展示 BOM 来源和展开时间 |
| 暂存区台账 | `配料记录` | 展示是否有暂存台账、入暂存区说明 |
| 仓库拣货进度 | `与配料记录关联的领料记录` | 展示待拣/已拣/已入暂存或已有领料记录 |
| 完成通知 | `配料记录` | 展示 CONFIRMED 后生成通知数量 |
| 分配回写 | `按任务查看配料情况` | 展示已回写任务数和待分配任务数 |

## 页面行为

详情页结构调整为：

1. 面包屑、标题、生产单 / 配料单、操作按钮。
2. Tab 区。
3. 当前 Tab 内容。

不再出现：

- Tab 上方四张 KPI 卡片。
- Tab 上方五张实现链路卡片。

## 验收标准

- 染色、印花、车缝、其他配料详情页不再渲染 Tab 上方摘要区。
- 裁片配料详情页仍保持当前已处理形态。
- 原摘要区关键信息能在对应 Tab 内找到。
- 不新增“摘要”“实现状态”“链路状态”等新 Tab。
- 页面中文文案不出现英文状态码。
- 现有配料详情入口、返回列表、关闭配料单、新增配料记录、领料记录 Tab 不失效。

## 验证

新增或扩展最小检查脚本，覆盖：

- 分类配料详情页 HTML 不包含 Tab 上方摘要区代表文案组合。
- 详情页仍包含 Tab。
- 迁移后的关键信息在对应 Tab 中可见。

继续运行：

- `npm run build`
- `npm run check:prototype-design-governance -- --all`
- 相关配料检查脚本。

## 不做

- 不改配料列表页顶部统计区。
- 不改配料、领料、退回状态模型。
- 不新增真实后端、接口、权限或数据库。
- 不做大规模页面模板重构。
