# 生产准备时效准备项依赖优化设计

## 背景

当前 `生产准备时效` 页面已经支持准备台账、确认工作项、准备项操作、本次用料、工作项上传和统计。但本轮核查发现，准备项之间的业务依赖还没有完整表达：

- 染色调色前，应先由跟单确认染色要求；当前系统把 `维护染色要求` 当成染色项旁边的附加按钮，不是正式准备工作项。
- 基码纸样、版衣制作、齐码纸样已有部分依赖数据，但操作栏没有把“前置未完成时不可操作”表达成稳定规则。
- 业务希望不可操作时直接置灰，不需要复杂等待原因。

本设计只修正主流程依赖关系，不讨论快捷路径，不新增真实权限系统、后端、审批流或复杂状态机。

## 目标

- 把 `确认染色要求` 纳入生产准备工作项，而不是染色项附加动作。
- 明确 `确认染色要求 -> 染色调色` 的前置依赖。
- 明确 `基码纸样 -> 版衣制作 -> 齐码纸样` 的前置依赖。
- 操作栏按依赖结果展示：可操作显示蓝色按钮，不可操作直接置灰。
- Mock 数据必须能覆盖依赖成立、依赖未完成、多个前置并行、已完成后的统计闭环。
- 校验脚本必须能防止未来 mock 数据再次破坏依赖关系。

## 非目标

- 不新增快捷发起路径。
- 不改变生产准备时效的菜单、路由和页面边界。
- 不引入真实权限、组织架构、后端接口或数据库。
- 不把染色生产加工单完成时间当作准备阶段染色调色完成时间。
- 不让一线员工理解完整依赖链，只在操作栏给出能不能点。

## 业务口径

### 染色要求

`确认染色要求` 是跟单要完成的准备工作项。它回答的是：染色团队按什么颜色、什么潘通色号、哪一种物料去打色或调色。

当商品需要纱线染色时，准备项应包含：

- `确认染色要求（纱线）`
- `染色调色（纱线）`

当商品需要面料染色时，准备项应包含：

- `确认染色要求（面料）`
- `染色调色（面料）`

依赖关系：

| 后续准备项 | 前置准备项 | 业务含义 |
| --- | --- | --- |
| 染色调色（纱线） | 确认染色要求（纱线） | 跟单先确认纱线颜色要求，染色团队才能上传纱线调色结果 |
| 染色调色（面料） | 确认染色要求（面料） | 跟单先确认面料颜色要求，染色团队才能上传面料调色结果 |

完成口径：

- `确认染色要求` 的完成时间是跟单确认染色要求的时间。
- `染色调色` 的完成时间是染色团队上传调色结果的时间。
- 两者都属于准备工作项，都会进入准备项完成情况和统计。
- 完成 `确认染色要求` 不等于完成 `染色调色`。

### 纸样与版衣

纸样链路按业务顺序推进：

| 后续准备项 | 前置准备项 | 业务含义 |
| --- | --- | --- |
| 版衣制作 | 基码纸样 | 先有基码纸样，车板团队才能做版衣 |
| 齐码纸样 | 版衣制作 | 版衣确认后，版师团队再做齐码纸样 |

如果同一记录同时存在梭织和毛织基码纸样：

- 版衣制作依赖所有已选基码纸样。
- 只完成其中一个基码，不允许提前做版衣。
- 齐码纸样依赖版衣制作完成。

## 页面行为

### 台账操作栏

未确认工作项：

- 只显示 `确认工作项`。
- 不显示准备项操作按钮。

已确认工作项：

- 显示 `查看详情`。
- 对每个已选准备项显示一个操作入口。
- 可操作：蓝色按钮。
- 不可操作：同一个动作文本置灰，不可点击。
- 已完成：不再显示上传动作，展示 `已完成` 或保留为不可重复操作状态。

不展示复杂等待原因，不展示长解释，不要求业务人员判断依赖链。

### 动作文案

| 准备项 | 可操作按钮 |
| --- | --- |
| 确认染色要求（纱线） | 确认纱线染色要求 |
| 染色调色（纱线） | 上传纱线调色结果 |
| 确认染色要求（面料） | 确认面料染色要求 |
| 染色调色（面料） | 上传面料调色结果 |
| 梭织基码纸样 | 上传梭织基码纸样 |
| 毛织基码纸样 | 上传毛织基码纸样 |
| 版衣制作 | 上传版衣结果 |
| 梭织齐码纸样 | 上传梭织齐码纸样 |
| 毛织齐码纸样 | 上传毛织齐码纸样 |

### 详情页

详情页需要能看清准备项依赖，不作为一线操作判断入口。

每个准备项展示：

- 准备项名称。
- 责任团队。
- 责任人。
- 当前状态。
- 计划完成时间。
- 实际完成时间。
- 前置准备项。
- 上传证据。

染色要求项额外展示：

- 对应物料图片、名称、编码、物料类型。
- 颜色名称。
- 潘通色号。
- 染色备注。
- 确认人和确认时间。

## Mock 数据要求

Mock 数据必须覆盖以下场景：

1. 面料染色要求已确认，面料染色调色可操作。
2. 面料染色要求未确认，面料染色调色置灰。
3. 纱线染色要求已确认，纱线染色调色可操作。
4. 纱线染色要求未确认，纱线染色调色置灰。
5. 染色要求已完成，染色调色已完成，统计都能计入。
6. 染色要求已完成，染色调色进行中，只计染色要求完成。
7. 梭织基码已完成，版衣制作可操作。
8. 梭织基码未完成，版衣制作置灰。
9. 毛织基码已完成，版衣制作可操作。
10. 梭织和毛织基码同时存在，必须两者都完成后版衣才可操作。
11. 版衣制作未完成，齐码纸样置灰。
12. 版衣制作已完成，齐码纸样可操作。
13. 跟单尚未确认工作项时，不出现任何准备项操作入口。
14. 已完成准备项不重复展示上传动作。

## 数据规则

准备项类型需要补充：

- `确认染色要求（纱线）`
- `确认染色要求（面料）`

责任团队：

- `确认染色要求` 归属跟单角色。
- `染色调色` 归属染色团队。

依赖字段：

- `染色调色（纱线）.dependsOnItemIds` 指向同记录的 `确认染色要求（纱线）`。
- `染色调色（面料）.dependsOnItemIds` 指向同记录的 `确认染色要求（面料）`。
- `版衣制作.dependsOnItemIds` 指向同记录所有已选基码纸样。
- `齐码纸样.dependsOnItemIds` 指向同记录的 `版衣制作`。

完成证据：

- 所有被判定为 `已完成` 的准备项必须有实际完成时间和上传或确认记录。
- 染色调色不能在对应染色要求未完成时进入 `已完成`。

## 校验要求

`scripts/check-production-preparation-timing.ts` 需要新增或调整以下校验：

- 所有已选 `染色调色（纱线）` 必须依赖 `确认染色要求（纱线）`。
- 所有已选 `染色调色（面料）` 必须依赖 `确认染色要求（面料）`。
- 染色调色已完成时，对应染色要求必须已完成。
- 染色要求未完成时，操作栏中的染色调色动作必须置灰。
- `版衣制作` 必须依赖同记录所有已选基码纸样。
- `齐码纸样` 必须依赖同记录的 `版衣制作`。
- 前置未完成时，后续操作必须置灰。
- 删除旧校验：`维护染色要求` 不再是附加动作，也不再排除在完成准备项统计外。

## 对抗式核查结论

### 数据完整性

反例：染色调色记录存在，但没有染色要求记录。

处理：mock 和脚本都必须阻断。只要出现染色调色，就必须有对应确认染色要求。

### 页面可见性

反例：业务人员只看到 `上传染色调色结果` 置灰，不知道是否漏了确认染色要求。

处理：台账不写长原因，但详情页必须展示前置准备项及其状态；管理人员可从详情判断卡点。

### 交互可达性

反例：染色要求作为附加按钮时，跟单不维护也能让染色团队上传结果。

处理：染色要求改成正式准备项，染色调色依赖它，前置未完成时上传动作置灰。

### 业务闭环

反例：跟单确认染色要求后，统计不计入，月度准备工作量少算。

处理：`确认染色要求` 作为准备项计入完成情况和统计；`染色调色` 仍按上传调色结果独立计入。

## 验收标准

- 页面操作栏中，所有受前置约束的准备项都按依赖结果蓝色可点或灰色不可点。
- 染色要求不再以 `维护染色要求` 附加按钮出现。
- 染色要求未完成时，染色调色上传动作不可点击。
- 基码纸样未完成时，版衣制作不可点击。
- 版衣制作未完成时，齐码纸样不可点击。
- mock 数据覆盖上述 14 个业务场景。
- `npm run check:production-preparation-timing` 能校验依赖规则。
- `npm run check:prototype-design-governance -- --all` 通过。
- `npm run build` 通过。

## 规格自检

- 无 `TODO`、`待定`、占位章节。
- 本规格只覆盖生产准备时效主流程依赖，不包含快捷路径。
- 染色要求的业务身份已明确：跟单准备工作项，不是染色项附加按钮。
- 纸样链路的前置关系已明确：基码纸样、版衣制作、齐码纸样顺序推进。
- 操作栏表达已明确：不可操作直接置灰，不做复杂等待原因。
