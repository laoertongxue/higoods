# 生产准备工作项要求维护设计

## 背景

当前 `生产准备时效` 页面已经具备准备台账、商品类型确认、准备项勾选、准备项独立操作、真实上传下载记录、产出展示和月度/明细统计。

本次需求来自 `智能纪要：版师需求表系统功能优化沟通会 2026年7月8日.docx`，核心不是新增完整权限系统，而是补齐跟单确认工作项时必须维护的前置要求，并让后续责任团队能明确看到自己该按什么要求执行。

已核查代码：

- `src/data/fcs/production-preparation-timing.ts`
- `src/data/fcs/production-preparation-timing-runtime.ts`
- `src/pages/production/preparation-timing.ts`
- `scripts/check-production-preparation-timing.ts`

当前代码现状：

- 准备项已有 `ownerTeam`、`ownerName`，但责任团队只是扁平字符串，没有团队和角色标签的关系。
- `确认工作项` 弹窗只保存商品类型、准备项和 `overrideReason`。
- 染色工作项已有独立操作入口，但没有颜色名称、潘通色号和对应面料要求。
- 花型项已有 `buyerReviewStatus`，但染色和印花买手审核不进入本次范围。

## 目标

- 在现有 `生产准备时效` 页面内增强确认工作项能力。
- 跟单确认工作项时，维护本次生产准备用的面料和做款/打板要求。
- 将原 `修正原因` 调整为通用备注，承载跟单补充说明。
- 为责任团队补充轻量角色标签映射，支撑页面表达和后续权限控制。
- 染色相关准备项提供独立的 `维护染色要求` 入口，由跟单填写颜色名称、潘通色号和对应面料。
- 保持染色团队上传调色结果和跟单维护染色要求两个动作分开。

## 非目标

- 不新增真实后端、数据库、文件服务或权限服务。
- 不做完整组织架构、部门、岗位和账号体系。
- 不新增买手审核流程卡点。
- 不把染色和印花的买手审核纳入本次开发。
- 不重构生产准备时效的统计口径。
- 不新建独立菜单或独立模块。

## 业务口径

### 角色和权限口径

会议纪要确认：当前系统暂未完成完整组织架构体系，权限匹配以角色标签为基础，不按部门维度划分。

本次只在原型中补充轻量配置，不做真实鉴权。

角色口径：

| 责任团队 | 允许操作角色标签 | 说明 |
| --- | --- | --- |
| 版师团队 | 版师、版师主管 | 操作梭织基码纸样、梭织齐码纸样 |
| 毛织团队 | 毛织版师、毛织主管 | 操作毛织基码纸样、毛织齐码纸样 |
| 车板团队 | 车版、车版主管 | 操作版衣制作 |
| 花型团队 | 花型师、花型主管 | 操作数码印/DTF/DTG花型 |
| 染色团队 | 染厂公共账号 | 上传染色调色结果 |
| 采购团队 | 采购、辅料采购 | 操作辅料下单 |
| 跟单角色 | 跟单 | 确认工作项、维护染色要求 |

规则：

- `确认工作项` 对所有跟单角色开放，不限制到单个跟单人员。
- 每个准备项的完成上传入口归属该准备项责任团队。
- `维护染色要求` 虽然挂在染色工作项下，但维护人是跟单角色。
- 角色映射只用于原型展示和后续权限边界，不在本次实现真实登录校验。

### 确认工作项

跟单进入 `确认工作项` 弹窗后，完成以下内容：

1. 确认商品类型。
2. 勾选准备项。
3. 选择本次使用面料。
4. 填写做款/打板要求。
5. 填写通用备注。

字段口径：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| 商品类型 | 是 | 沿用现有四类商品准备类型 |
| 准备项 | 是 | 系统按商品类型带出默认项，跟单确认选填项 |
| 本次用料 | 是 | 跟单选择或填写面料名称/面料编号 |
| 做款/打板要求 | 是 | 给版师、车板、毛织等团队的执行要求 |
| 通用备注 | 否 | 替代现有 `修正原因`，用于补充说明 |

说明：

- 商品类型如果从系统推导改成人工修正，通用备注中应写明原因。
- 辅料下单仍按现有规则可取消勾选。
- 染色潘通信息不放在 `确认工作项` 弹窗。

### 染色要求维护

当跟单选择了以下任一准备项时，操作栏应出现独立入口 `维护染色要求`：

- `染色调色（纱线）`
- `染色调色（面料）`

该入口归属于对应染色工作项，但由跟单维护。

字段口径：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| 对应面料 | 是 | 从确认工作项的本次用料中选择或复用 |
| 颜色名称 | 是 | 例如雾蓝、浅灰、米白 |
| 潘通色号 | 是 | 例如 `PANTONE 14-4318 TPX` |
| 染色备注 | 否 | 特殊色差、色牢度、样卡等补充要求 |

规则：

- 染色要求未维护时，染色工作项展示 `待跟单维护染色要求`。
- 染色要求维护后，染色团队才能按要求上传调色结果。
- 染色团队上传结果仍走现有工作项上传弹窗。
- 维护染色要求不等于完成染色工作项，不计入月度完成数量。
- 染色工作项完成数量仍以实际完成上传时间为准。

### 买手审核

本次不新增买手审核流程。

规则：

- 保留现有花型项的买手确认展示。
- 不新增染色买手审核。
- 不新增印花买手审核。
- 不让买手审核影响准备项确认、染色要求维护或染色团队上传。

## 页面行为

### 准备台账操作栏

未确认工作项时：

- 显示 `确认工作项`。
- 不显示各团队准备项操作入口。
- 不显示 `维护染色要求`。

已确认工作项后：

- 显示 `查看详情`。
- 按已选择准备项显示独立操作入口。
- 对已选择染色项，显示 `维护染色要求`。
- 染色上传入口和 `维护染色要求` 分开显示。

### 详情抽屉

详情抽屉只查看完整信息，不承载提交动作。

需要新增展示：

- 本次用料。
- 做款/打板要求。
- 通用备注。
- 责任团队和角色标签说明。
- 染色要求维护状态。
- 已维护的颜色名称、潘通色号、对应面料。

### 工作项卡片

工作项卡片继续展示责任团队、责任人、计划时间、实际完成、上传记录和下载记录。

染色工作项额外展示：

- 染色要求状态。
- 对应面料。
- 颜色名称。
- 潘通色号。
- 染色备注。

## 数据设计

### 生产准备记录

在准备记录上补充跟单确认阶段的前置要求：

```ts
materialRequirement: {
  materialNo: string
  materialName: string
}
sampleRequirementText: string
confirmationRemark: string
```

说明：

- `materialRequirement` 表示本次准备使用的面料。
- `sampleRequirementText` 表示做款/打板要求。
- `confirmationRemark` 替代现有 `prepTypeOverrideReason` 的页面含义。

### 责任团队角色

新增轻量配置：

```ts
interface PreparationOwnerRoleRule {
  ownerTeam: string
  roleLabels: string[]
  actionScope: string
}
```

说明：

- 不绑定真实账号。
- 不接真实权限。
- 用于页面展示、自检脚本和后续权限扩展。

### 染色要求

在染色准备项上补充：

```ts
dyeRequirement?: {
  materialNo: string
  materialName: string
  colorName: string
  pantoneCode: string
  remark: string
  maintainedBy: string
  maintainedAt: string
}
```

说明：

- 仅染色工作项使用。
- `maintainedBy` 固定表达为跟单维护人。
- `maintainedAt` 是维护时间，不是染色完成时间。

## Mock 数据要求

需要补齐以下场景：

1. 纯梭织记录：已确认工作项，填写面料和打板要求，未选择染色。
2. 毛织记录：已确认工作项，选择染色面料项，已维护潘通色号，染色待上传。
3. 毛织&梭织记录：同时选择纱线染色和面料染色，两个染色项分别维护不同潘通色号。
4. 烫画&直喷记录：只有花型项，无染色要求入口。
5. 未确认记录：不出现染色要求入口，也不出现团队上传入口。
6. 染色要求未维护记录：染色项显示 `待跟单维护染色要求`。

Mock 数据必须保持业务一致：

- 未确认工作项的记录不得有生产单、产出对象或已完成准备项。
- 染色要求维护时间不得被当作准备项完成时间。
- 已完成染色项必须有真实上传记录。
- 未选择染色项不得展示潘通字段。

## 实施边界

本设计只要求在现有字符串模板和本地 runtime 中表达业务链路。

最小实现应优先复用：

- 现有 `确认工作项` 弹窗。
- 现有准备项独立操作入口。
- 现有 runtime localStorage。
- 现有上传记录和下载记录结构。
- 现有统计口径。

不为本次需求新增复杂权限框架、表单引擎或状态机。
