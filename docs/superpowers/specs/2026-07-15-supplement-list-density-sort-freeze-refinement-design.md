# 补料管理标准列表紧凑度、排序图标与冻结列修订设计

## 1. 背景与目标

补料管理已经作为标准列表页模板，但现场查看发现三处需要收口：

1. 摘要卡片内部标签与数值分成两行，垂直占用过高。
2. 可排序列在表格局部刷新后缺少稳定可见的排序图标。
3. 用户选择冻结的列没有从横向滚动开始就固定在最左侧，不符合“冻结列”的使用预期。

本次只修订标准列表公共组件和补料管理模板页面，不迁移其他现有列表页。

## 2. 已确认的视觉方案

采用 B「平衡紧凑」方案：

- 摘要卡片高度为 48px。
- 每张卡片内部只有一行：标签在左、数值在右。
- 表头目标高度约 42px，保留低分辨率办公电脑上的可点击性。
- 冻结区右边缘和操作列左边缘使用轻微阴影或分隔线，明确固定区边界。

## 3. 摘要卡片

标准列表摘要组件继续按多个独立卡片展示，不改统计数据口径。

每个卡片必须满足：

- 标签和数值在同一横排，垂直居中。
- 卡片高度固定为 48px，不再使用上下两段布局。
- 标签使用次要文字色；数值使用加粗、等宽数字。
- 在 1280×720 下三个现有摘要卡片保持同一行；空间不足时才允许按组件既有响应规则换行。

## 4. 排序图标与交互

所有 `sortable` 列必须显示明确排序图标：

- 未排序：双向箭头。
- 升序：向上箭头。
- 降序：向下箭头。

交互规则：

- 列标题和图标属于同一个按钮，均可点击。
- 状态循环保持“升序 → 降序 → 恢复默认”。
- `aria-sort` 和按钮的中文 `aria-label` 必须与当前及下一状态一致。
- 图标由标准列表组件直接输出内联 SVG，不依赖整页 Lucide 扫描，避免表格局部刷新后图标消失。
- 图标区域为 24×24px，既能明确表达可排序，也不增加表头高度。

## 5. 冻结列行为

冻结列必须表现为真正的左侧固定列：

- 用户勾选冻结后，该列立即进入表格最左侧固定区。
- 从横向滚动开始到结束，该列的左侧坐标保持不变。
- 多个冻结列按当前用户列顺序从左到右排列。
- 冻结列之后才渲染普通可滚动列；操作列始终固定在最右侧。
- 取消冻结后，该列回到用户配置的普通列顺序。
- 冻结偏好继续按路由持久化，刷新页面后保持。
- 继续执行冻结总宽度上限，避免固定区挤占 1280×720 下的中间可滚动区域。
- 冻结区最右侧显示轻微阴影或边线，横向滚动时仍能识别固定区边界。

冻结只改变渲染分组，不改写用户保存的列顺序；因此取消冻结后可以无损恢复原位置。

## 6. 组件边界

- `src/components/ui/list-page.ts`：负责标准摘要卡片的单行紧凑布局。
- `src/components/ui/list-table.ts`：负责排序 SVG、冻结列分组、左右 sticky 定位和固定区边界。
- `src/pages/process-factory/cutting/supplement-management.ts`：继续提供业务列定义、冻结宽度上限和数据，不复制公共布局实现。
- 补料管理之外的现有页面不在本次调整范围内；后续新增或调整列表页通过公共组件自动采用修订后的标准。

## 7. 验收与回归

自动化验收至少覆盖：

1. 摘要卡片高度为 48px，标签和值同一横排。
2. 三种排序状态均有可见的内联 SVG 图标，表格局部刷新后仍存在。
3. 排序三态、`aria-sort`、数据顺序保持正确。
4. 冻结一个非首列后，它立即位于左侧固定区；横向滚动前后左坐标误差不超过 1px。
5. 多列冻结时按照用户列顺序排列，且彼此不覆盖。
6. 取消冻结后列回到原普通顺序。
7. 操作列横向滚动前后仍贴合容器右边缘。
8. 1366×768 和 1280×720 下页面主体无横向溢出，表格内部可横向滚动。
9. 所有相关交互继续局部刷新，`main` 节点不替换，控制台无错误。

## 8. 非目标

- 不调整补料业务字段、筛选条件、分页口径或 Mock 数据。
- 不迁移其他历史列表页。
- 不引入新的 UI 框架或全局状态体系。
- 不改变已确认的列偏好持久化范围。
