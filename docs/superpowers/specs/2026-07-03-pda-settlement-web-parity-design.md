# PDA 结算与 Web 对账数据同源设计

日期：2026-07-03
状态：已确认设计，待书面规格审查
适用项目：HiGood FCS 原型仓库

## 背景

当前 PDA 端结算页与 Web 端对账单生成页存在明显口径错位。

Web 端 `新建对账单` 的对象反查演示数据在 `src/pages/statements.ts` 内部生成，包括 `MOCK-SETTLE-PO-001`、`MOCK-SETTLE-PO-002`、`MOCK-SETTLE-PO-003`、15 条预结算流水和 9 张后道质检单。PDA 端结算页不读取这组页面内数据，而是通过共享种子和预结算流水仓库聚合 `ST-LINK-*`、`PSL-TASK-*`、`QDL-*` 数据。

因此，Web 端看到的“按生产单反查、质检事实、扣款明细”和 PDA 端看到的“结算周期、正式流水、对账与预付款”不是同一批对象。

## 目标

本次调整的目标是让 PDA 结算成为 Web 对账与结算链路的移动端查看和反馈视图，而不是独立的结算计算器。

必须满足：

- Web 和 PDA 使用同一批预结算流水、生产单反查结果、后道质检事实和返工反扣流水。
- PDA 金额以已生成对账单为主，不把整周期流水池金额误显示为对账单金额。
- 未进入对账单的预结算流水可以展示，但必须标记为待入对账，不能混入当前对账单净额。
- 工厂端 PDA 保留查看、确认、申诉、查看结算资料、查看付款结果能力，不提供扣款金额编辑。
- 当前金额统一使用 IDR。

## 非目标

- 不重做真实后端、权限系统、数据库或 API。
- 不改工厂档案完整结算资料模型。
- 不新增审核流程。
- 不把 PDA 做成完整后台工作台。
- 不重新设计整个预付款批次页面。

## 当前问题

### 1. Web 对象反查 mock 没有进入共享数据层

Web 页面内的 `MOCK-SETTLE-*` 数据只服务 `statements.ts`，PDA 端无法追溯这些生产单、流水和质检事实。

### 2. PDA 混用了周期流水池金额和对账单金额

例如 `双周 2026-03-15 ~ 2026-03-31`：

- 周期流水池：收入 `121,550 IDR`，扣款 `1,320 IDR`，净额 `120,230 IDR`。
- 当前对账单 `ST-LINK-2026-0006`：只有 1 条收入流水，净额 `121,550 IDR`。

PDA 首屏展示了周期净额，但同一张卡又展示对账单状态，业务上容易被理解为这张对账单净额。

### 3. PDA 仍以结算周期作为主对象

业务口径已经调整为 `工厂 + 自定义时间段 + 结算对象`。结算周期只作为参考和默认建议，不应该成为 PDA 金额主对象。

### 4. PDA 质量影响区存在过滤字段错误

`renderQualityImpactOverview` 使用 `summary.settlementPartyId` 过滤质量扣款，但 `SettlementCycleSummary` 当前没有该字段，运行时会变成 `undefined`，导致质量影响概况与正式扣款流水不一致。

## 设计原则

### 1. 对账单是财务主对象

PDA 展示金额时优先读 `StatementDraft`：

- `totalEarningAmount`
- `totalDeductionAmount`
- `netPayableAmount`
- `items`
- `settlementRangeStartAt`
- `settlementRangeEndAt`
- `settlementObjectMode`

如果没有对账单，PDA 只能展示待入对账流水，不展示“本期净额”作为财务结果。

### 2. 结算周期只做分组和参考

周期仍可用于列表分组、默认日期、计划预付款日参考，但页面文案不能让工厂误以为周期就是正式对账对象。

### 3. 预结算流水和质检事实必须共享

Web 新建对账单使用的数据应下沉到 `src/data/fcs/`，通过 selector 输出：

- 可入对账的预结算流水。
- 生产单反查结果。
- 生产单完成判断。
- 后道质检事实。
- 按生产单和瑕疵原因汇总的扣款事实。
- 返工接收对象为后道工厂的返工反扣流水。

Web 和 PDA 都只能消费这套共享输出，不再各自造一套演示数据。

## 数据设计

### 共享演示数据

新增或调整一个共享数据模块，承载当前 Web 对象反查需要的演示数据：

```text
3 张生产单
每张生产单 5 条预结算流水
每张生产单 3 张后道质检单
每张质检单 3 个 SKU
每个 SKU 有瑕疵数量和瑕疵原因
每张质检单都有返工
每张质检单 2 个 SKU 返工到原工厂，1 个 SKU 返工到后道工厂
```

共享输出需要能被 Web 和 PDA 同时读取。

### 对账单与流水关系

生成对账单后，PDA 应按 `StatementDraft.items` 展示已入单明细：

- 任务收入流水。
- 返工反扣流水。
- 对账单内手工填写的瑕疵扣款。
- 对账单内手工填写的延误扣款。

未被 `StatementDraft.itemSourceIds` 引用的流水，展示为待入对账。

### PDA 列表对象

PDA 结算列表建议展示两类卡片：

1. 对账单卡片：已经生成对账单的正式财务对象。
2. 待入对账卡片：当前工厂和时间段内仍未进入对账单的预结算流水提示。

对账单卡片金额只来自对账单。待入对账卡片不显示正式净额，只显示待入流水数量和金额参考。

## PDA 页面设计

### 结算列表

列表主卡应改为对账单视角：

- 对账单号。
- 工厂。
- 自定义时间段。
- 结算对象：按生产单 / 按预结算流水。
- 应付、扣款、净额。
- 对账单状态。
- 工厂反馈状态。
- 预付款批次状态。
- 飞书付款状态。
- 打款回写状态。

如果当前周期有多张对账单，展示多张对账单卡片，而不是合并成一个周期金额。

### 对账单详情

详情页保留现有四个页签，但调整口径：

- 总览：读对账单金额和状态。
- 质检扣款：展示本对账单已引用的质检扣款明细，以及仍待处理的质检扣款提醒。
- 正式流水：默认只展示本对账单 items 对应流水，可切换查看同范围待入对账流水。
- 对账与预付款：展示对账单、预付款批次、飞书付款和打款回写。

### 待入对账流水

待入对账流水必须明确说明：

```text
这些流水尚未进入当前对账单，仅供业务人员在 Web 端生成或重开对账单时处理。
```

PDA 端不提供加入对账单操作。

## Web 页面设计

Web `新建对账单` 继续保留当前流程：

- 按预结算流水：基础范围 -> 预结算流水 -> 质检扣款 -> 金额确认。
- 按生产单：基础范围 -> 对象反查 -> 质检扣款 -> 金额确认。

但对象反查、质检事实和返工反扣数据必须来自共享 selector，不能由页面局部构造。

## 验证要求

最小验证需要覆盖：

1. Web 对象反查可见 `MOCK-SETTLE-PO-001/002/003`。
2. PDA 同一工厂、同一时间段能追溯到同一批生产单或其生成后的对账单明细。
3. PDA 对账单金额等于 `StatementDraft` 金额。
4. PDA 不再把周期流水池净额显示成对账单净额。
5. PDA 质量影响区过滤使用明确工厂 ID，不使用不存在的 `summary.settlementPartyId`。
6. Web 和 PDA 都只显示 IDR。

## 实施边界

推荐采用小步实现：

1. 先抽共享数据源。
2. 再让 Web 读取共享数据源。
3. 再调整 PDA 列表和详情金额口径。
4. 最后补检查脚本。

不做全局领域模型重构。
