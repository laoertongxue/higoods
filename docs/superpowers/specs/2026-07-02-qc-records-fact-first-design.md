# 质检记录事实优先设计

## 背景

当前 `质检记录` 页面同时承载质检事实、责任响应、异议处理、来源反扣和对账提示，页面主视线偏财务处理，业务人员难以快速核对“质检到底发生了什么”。

代码核查发现：

- 后道工厂 `质检单` 源数据已经有 SKU 级返工事实，包括 `reworkQty`、`reworkReceiveFactoryName`、`defectAcceptedQty`、`defectReasonItems`、`qcSkuResults` 和 `sourceChargeback`。
- 当前 `质检记录` 列表主要来自质量扣款/回货质检兼容视图，只保留 `质检数量 / 合格数量 / 不合格数量 / 责任数量 / 扣款链路` 等字段，没有完整带入后道质检单的返工事实。
- 用户已确认目标：整个 `质检记录` 页面统一改成“质检事实优先、扣款/对账收起”的结构，扣款/对账只需要弱化展示进入了哪个对账单、对应哪条扣款明细。

## 目标

`质检记录` 默认只回答三件事：

1. 这张质检记录的来源、对象、时间和结果是什么。
2. 质检数量、合格数量、返工数量、瑕疵数量分别是多少，返工接收对象是谁。
3. 如果已经进入结算链路，它进入了哪个对账单、对应哪条扣款明细。

## 非目标

- 不在 `质检记录` 内编辑扣款金额、扣款决策或对账单。
- 不把质量扣款工作台、异议处理、对账单生成逻辑迁入 `质检记录`。
- 不新增复杂状态机或真实后端。当前项目仍按原型仓库处理。
- 不扩大到后道质检单本身的录入结构，除非实现时发现字段读取有缺口。

## 信息架构

### 列表页

主列保留：

- 质检单号
- 来源类型
- 生产单 / SKU
- 质检数量
- 合格数量
- 返工数量
- 瑕疵数量
- 返工接收对象
- 质检结果
- 质检时间
- 对账追溯
- 操作

弱化或移除主列：

- 责任状态
- 工厂响应
- 异议状态
- 来源反扣金额
- 对账提示状态
- 大段财务说明

`对账追溯` 只展示低权重文本：

- 已进入对账：`对账单：<statementNo> / 扣款明细：<deductionLineNo>`
- 未进入对账：`未进入对账`
- 仅有来源反扣事实但未入账：`待对账引用`

不在列表显示金额。

### 详情页

详情页顺序固定为：

1. 基本事实：质检单号、来源类型、生产单、来源工厂、接收方、质检台、质检人、质检时间、质检结果。
2. 数量事实：质检、合格、返工、瑕疵；返工按接收对象分组展示。
3. SKU 级明细：SKU、颜色/尺码、质检数量、合格数量、返工数量、返工接收对象、瑕疵数量、瑕疵原因、后道项目。
4. 质检证据：图片、视频、文件等现场证据。
5. 对账追溯：只读弱化展示对账单号和扣款明细号；不展示金额和裁决过程。
6. 原始链路信息：折叠展示来源单据、任务、回货批次、旧兼容字段。

## 数据来源与映射

### 统一事实视图

新增或调整一个页面级事实视图，聚合两类来源：

- 现有质量链路：`listPlatformQcListItems()` / `getPlatformQcDetailViewModelByRouteKey()`。
- 后道质检单：列表使用 `listPostFinishingQcOrders()`，需要实体状态时使用 `listPostFinishingQcOrderEntities()`。

该事实视图只服务 `质检记录` 页面展示，不改变原始领域数据。

### 后道质检单字段

后道质检单进入 `质检记录` 时必须保留：

- `qcSkuResults[].reworkQty`
- `qcSkuResults[].reworkReceiveFactoryName`
- `qcSkuResults[].defectAcceptedQty`
- `qcSkuResults[].defectReasonItems`
- `qcSkuResults[].postProjectJudgements`
- `qcSkuResults[].sourceChargeback`

`sourceChargeback` 只作为“来源反扣事实”参与追溯判断，不在质检记录主视图展示金额。

### 对账追溯字段

事实视图提供一个简化追溯对象：

```ts
type QcSettlementTrace = {
  statementNo?: string
  deductionLineNo?: string // 可用现有扣款依据编号或对账单内扣款明细编号
  statusLabel: '未进入对账' | '待对账引用' | '已进入对账'
}
```

页面只显示这个对象，不显示扣款金额、争议处理、响应截止、正式流水状态。

## 交互原则

- 列表默认一眼看到质检事实，不需要横向滚动。
- 返工数量为 0 时，返工接收对象显示 `—`。
- 多个返工接收对象时，列表显示聚合文本，详情按对象分组。
- 扣款/对账信息使用灰色小字或折叠区，不能使用红色、橙色大面积强调。
- 操作只保留 `查看详情` 和必要的来源单据跳转。

## 成功标准

- 后道质检单中有返工时，`质检记录` 列表和详情都能看到返工数量与返工接收对象。
- 详情能看到 SKU 级返工、瑕疵数量和瑕疵原因。
- 页面不再把扣款、异议、工厂响应作为主流程展示。
- 扣款/对账只显示对账单号和扣款明细号；无编号时显示明确空态。
- 现有回货质检、后道复检、旧兼容质检记录仍能展示基本事实。

## 验证

最小验证范围：

- 跑现有后道质检检查脚本，确认后道质检单返工事实仍完整。
- 增加或调整一个轻量检查脚本，断言 `质检记录` 页面包含返工数量、返工接收对象、SKU 级瑕疵原因，并且列表主视图不再出现 `工厂响应`、`异议状态`、`来源反扣金额` 等主列文案。
- 构建通过。
- 浏览器打开 `质检记录` 页面，核对列表和一个后道质检详情。

## 范围控制

优先在 `质检记录` 页面和页面级数据适配处完成，不重构质量扣款模块，不改后道质检录入流程，不新增全局状态层。
