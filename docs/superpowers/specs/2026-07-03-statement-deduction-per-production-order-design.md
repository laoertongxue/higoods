# 对账单扣款按生产单拆分设计

日期：2026-07-03
状态：已通过口头设计评审，等待书面规格审查
适用项目：HiGood FCS 原型仓库

## 背景

当前 `新建对账单 -> 质检扣款` 页面把瑕疵原因按整张对账单汇总展示，例如 `做工原因` 一行关联多张生产单并填写一个扣款金额。

这个口径不严谨：不同生产单可能对应不同款式、不同成本、不同扣款标准。即便瑕疵原因相同，扣款金额也可能不同。返工反扣和延误扣款也存在同样问题，必须能落到明确生产单。

## 目标

本次调整只解决对账单生成时的扣款录入粒度：

- 瑕疵扣款金额按 `生产单 + 瑕疵原因` 填写。
- 返工反扣列表展示生产单。
- 延误扣款按生产单展示时间辅助信息，并按生产单填写金额和说明。
- 金额确认按明细汇总到整张对账单净额。

## 非目标

- 不细化到 `SKU + 瑕疵原因` 金额录入；用户已确认 `生产单 + 瑕疵原因` 足够。
- 不新增审核流程。
- 不改质检记录事实页。
- 不改后道质检单录入模型。
- 不引入真实后端、权限或复杂状态管理。

## 页面结构

### 质检扣款

页面按生产单分组展示：

```text
生产单 A
  做工原因  数量 X  扣款金额  扣款说明
  脏污      数量 X  扣款金额  扣款说明

生产单 B
  做工原因  数量 Y  扣款金额  扣款说明
```

数量来自已纳入本期对账对象关联的质检事实。只展示车缝工厂责任原因：

```text
做工原因、脏污、抽纱、做错、做毁、破洞
```

全局原因汇总可以保留为辅助阅读，但不能作为填钱入口。

### 返工反扣

返工反扣表增加生产单列：

```text
生产单 / 流水号 / 质检记录 / 返工数量 / 扣款金额
```

返工反扣金额来自预结算流水，不在这里编辑。

### 延误扣款

延误扣款按生产单展示：

```text
生产单 / 开始时间参考 / 最后交出时间 / 延误扣款金额 / 延误说明
```

系统只提供时间辅助业务人员判断，不自动计算延误扣款。

## 数据口径

### 瑕疵扣款输入状态

页面状态从按原因保存：

```ts
Record<reasonName, { amount, remark }>
```

调整为按生产单和原因保存：

```ts
Record<productionOrderNo, Record<reasonName, { amount, remark }>>
```

生成对账单明细时，每个有金额的 `生产单 + 原因` 生成一条手工瑕疵扣款明细。

### 延误扣款输入状态

页面状态从整张单一笔延误扣款：

```ts
{ amount, remark }
```

调整为按生产单保存：

```ts
Record<productionOrderNo, { amount, remark }>
```

每个有金额的生产单生成一条延误扣款明细。

### 金额确认

金额确认汇总：

```text
任务收入流水
- 返工反扣流水
- 生产单瑕疵扣款
- 生产单延误扣款
= 本期应付净额
```

明细行必须保留生产单号、扣款类型、原因或说明、数量和金额。

## 交互原则

- 默认展示生产单分组，业务人员先看生产单，再填扣款。
- 同一生产单内只展示有瑕疵数量的原因，避免空行过多。
- 金额输入仍使用 IDR，当前不做多币种。
- 输入时不触发整页重绘，保持现有轻交互方式。

## 验证

最小验证范围：

- 检查脚本断言瑕疵扣款输入存在 `productionOrderNo + reasonName` 粒度。
- 检查脚本断言返工反扣表包含生产单列。
- 检查脚本断言延误扣款按生产单展示时间和金额输入。
- `npm run check:statements`
- `npm run check:factory-settlement-statements`
- `npm run build`
- 浏览器验证 `新建对账单 -> 质检扣款`：每张生产单能分别填写同一原因的不同扣款金额。
