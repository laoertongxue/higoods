# 工厂结算规则产品设计方案

日期：2026-07-01
状态：已通过口头设计评审，等待书面规格审查
适用项目：HiGood FCS 原型仓库

## 1. 背景与目标

本方案来自 2026 年 7 月 1 日“工厂结算规则相关事宜沟通会”、后续多轮业务澄清，以及当前项目代码核查。

本次目标不是做一个全自动财务系统，而是在现有原型里建立清晰、可解释、可审计的工厂结算工作台：

- 业务人员能按自定义时间段生成对账单。
- 业务人员能选择按预结算流水结算，或按生产单结算。
- 按生产单结算时，系统必须全部展示反查出的生产单，并清楚标记哪些可进入本期对账，哪些未完成且差多少。
- 质检事实、扣款确认、财务付款必须分层，避免质检记录直接变成扣款单。
- 所有金额必须具备币种、来源、确认人和锁账状态，满足财务会计追溯要求。

## 2. 当前代码相关范围

本方案涉及以下现有页面和数据模块：

- 工厂档案结算信息：`src/pages/settlement/*`
- 预结算流水：`src/data/fcs/pre-settlement-ledger-repository.ts`
- 对账单：`src/pages/statements.ts`、`src/data/fcs/store-domain-statement-source-adapter.ts`
- 预付款批次：`src/pages/batches.ts`、`src/data/fcs/store-domain-settlement-types.ts`
- 工厂端 PDA 结算：`src/pages/pda-settlement.ts`
- 后道工厂质检单：`src/data/fcs/post-finishing-domain.ts`、`src/pages/process-factory/post-finishing/qc-orders.ts`
- 质检记录：`src/pages/qc-records/*`
- 扣款分析：`src/pages/deduction-analysis.ts`、`src/data/fcs/quality-deduction-*`
- 生产单与交出数量来源：生产单、裁片完成、交出记录、后道质检记录相关 mock/domain 数据

当前代码的主要冲突点：

- 对账单生成仍强依赖“工厂 + 结算周期”，但业务确认实际按自定义时间段生成。
- 预结算流水仍带有结算周期字段，并按周期过滤；会议待办明确提出后续应移除流水上的结算周期字段。
- 后道质检单和质检记录中存在扣款决策或扣款金额展示，和“质检记录只展示事实、不承载扣款决策”的新口径冲突。
- 扣款分析当前偏向从质检/扣款依据链路分析，应改为以对账单中业务确认的数据为财务口径来源。

## 3. 核心业务原则

### 3.1 工厂档案只提供默认建议

工厂档案里的结算周期保留，但只作为默认建议和后续治理目标，不作为当前生成对账单的硬限制。

工厂档案仍维护：

- 默认结算周期
- 默认币种，默认 `IDR`
- 收款账户
- 默认扣款规则
- 结算资料生效版本

创建对账单时可以带出默认周期建议和默认币种，但业务人员最终以自定义时间段生成。

### 3.2 结算对象由业务人员选择

业务人员创建对账单时必须选择结算对象：

- 按预结算流水结算
- 按生产单结算

两种方式都基于同一个自定义时间段。区别在于：

- 按预结算流水：业务人员直接选择时间段内符合条件的流水。
- 按生产单：系统先用时间段内的预结算流水反查生产单，再对生产单做完整性判断。

### 3.3 未完成生产单整张不进本期对账

按生产单结算时，生产单只要未完整，整张生产单不进入本期对账，不做部分进入。

系统仍必须展示所有反查出来的生产单，让业务人员知道系统为什么未纳入。

展示状态：

- 已完成，可纳入本期对账
- 未完成，不纳入本期对账，差 X 件

## 4. 生产单完整性判断

生产单完整性按数量口径判断：

```text
结算口径累计交出总数量 == 裁片完成数量
```

差额计算：

```text
差额 = 裁片完成数量 - 结算口径累计交出总数量
```

结算口径累计交出总数量需要考虑后道工厂质检单中的返工接收对象：

- 返工接收对象是原工厂：返工数量不重复累加到原工厂累计交出量。
- 返工接收对象是后道工厂：返工数量计入累计交出总量，用于让生产单数量和裁片完成数量对齐。
- 后道工厂接收返工数量虽然计入完成判断，但金额上必须从车缝工厂应付中扣回。

每张反查生产单必须展示详细数量信息：

- 裁片完成数量
- 常规累计交出数量
- 结算口径累计交出总数量
- 差额
- 累计返工数量，按原工厂接收和后道工厂接收区分
- 瑕疵总数量
- 车缝工厂责任瑕疵数量
- 各瑕疵原因数量汇总
- 交出记录维度明细

交出记录维度明细至少包括：

- 交出记录号
- 交出时间
- 对应生产单
- 交出工厂
- 接收对象
- 交出数量
- 质检数量
- 合格数量
- 返工数量
- 返工接收对象
- 瑕疵数量
- 瑕疵原因汇总
- 来源后道质检单

## 5. 车缝工厂责任原因

属于车缝工厂责任的瑕疵原因固定为：

```text
做工原因、脏污、抽纱、做错、做毁、破洞
```

这些原因进入车缝工厂责任统计，并可在对账单中生成扣款建议。

其他瑕疵原因仍作为质检事实展示，但默认不进入车缝工厂责任扣款。业务人员如需扣款，必须在对账单中手工填写原因和金额。

## 6. 质检事实与扣款边界

### 6.1 后道工厂质检单

后道工厂质检单是质检事实来源，可以记录：

- 质检数量
- 合格数量
- 返工数量
- 返工接收对象
- 瑕疵数量
- 瑕疵原因
- 后道项目判断
- 来源反扣金额

后道质检单中可以存在反扣扣款，因为它是现场事实的一部分。但该金额只是来源事实或扣款建议，不代表已进入财务结算。

### 6.2 质检记录

质检记录只展示质检事实，不允许承载独立扣款决策。

质检记录可以展示：

- 来源质检单存在反扣
- 来源反扣金额
- 来源质检单号和明细

质检记录不允许：

- 编辑扣款金额
- 编辑扣款决定
- 生成正式扣款流水
- 修改对账单金额

### 6.3 对账单

扣款真正生效的位置是对账单。

业务人员在对账单里确认或填写：

- 瑕疵原因扣款
- 后道工厂接收返工反扣
- 延误扣款
- 不扣款原因
- 币种
- 金额
- 备注

只有对账单确认后的扣款才进入扣款分析和预付款净额。

## 7. 对账单生成流程

新建对账单采用三步：

### 步骤 1：选择范围

业务人员选择：

- 工厂
- 开始日期
- 结束日期
- 结算对象：预结算流水 / 生产单
- 币种：默认 `IDR`，可选 `CNY`、`USD`
- 备注

工厂档案中的结算周期可作为推荐日期范围，但不强制。

### 步骤 2：查看反查结果

按预结算流水：

- 展示时间段内可进入对账单的流水。
- 显示已入单、已入批次、不可重复选择等状态。

按生产单：

- 展示时间段内预结算流水反查出的全部生产单。
- 标记完成状态。
- 展示差额和完整数量明细。
- 默认只选择已完成生产单。
- 未完成生产单不可强制纳入，只允许备注说明。

### 步骤 3：确认金额

对账单草稿分区展示：

- 应付加工费
- 质量扣款
- 后道返工反扣
- 延误扣款
- 其他调整
- 本期应付净额

生成对账单时必须冻结以下快照：

- 工厂
- 自定义结算时间段
- 结算对象
- 工厂结算资料版本
- 收款账户快照
- 币种
- 纳入的流水或生产单
- 生产单完成判断明细
- 质检事实来源
- 业务确认扣款
- 创建人和创建时间

## 8. 扣款分析

扣款分析的数据源改为对账单中业务确认的数据。

分析对象：

- 质量扣款
- 后道返工反扣
- 延误扣款
- 其他调整

分析维度：

- 工厂
- 生产单
- 对账单
- 时间段
- 瑕疵原因
- 责任类型
- 扣款类型
- 币种
- 对账单状态
- 是否进入预付款批次

扣款分析不再把未确认的质检记录金额当成财务金额。未进入对账单的质检事实只能作为来源证据，不计入扣款统计。

## 9. 预结算流水

预结算流水是对账单的候选来源。

新口径下：

- 流水不再以结算周期作为生成对账单的主筛选条件。
- 流水必须支持按工厂、发生时间、生产单、状态、类型过滤。
- 流水可继续保留旧的周期展示字段作为历史兼容，但不能作为本次对账单生成的硬条件。
- 已进入对账单或预付款批次的流水不能重复进入新对账单。

流水类型至少包括：

- 任务收入流水
- 对账单确认后的质量扣款流水
- 对账单确认后的后道返工反扣流水
- 对账单确认后的延误扣款流水
- 其他调整流水

## 10. 预付款批次

预付款批次只消费已达到可付款条件的对账单，不重新计算结算口径。

入批条件：

- 对账单已生成
- 工厂已确认，或平台已按规则代确认
- 无未处理申诉
- 金额完整
- 币种完整
- 收款账户快照完整
- 未被其他预付款批次占用

进入预付款批次后，对账单金额锁定。

如后续发现错误，不反改历史批次，必须通过：

- 冲销
- 补扣
- 补付
- 下期调整

## 11. PDA 结算

PDA 面向工厂和现场人员，定位为查看、确认、申诉和收款结果消费端，不承担平台核算。

PDA 可展示：

- 本期对账单
- 应付金额
- 扣款金额
- 本期净额
- 扣款原因
- 来源质检单
- 预付款状态
- 打款结果

PDA 可操作：

- 确认对账单
- 发起申诉
- 查看申诉处理结果
- 查看收款信息

PDA 不允许：

- 修改扣款金额
- 修改生产单完成判断
- 修改币种
- 修改预付款批次

交互要求：

- 数字优先，先展示“应付、扣款、净额”。
- 文案短句化，避免抽象词。
- 操作按钮使用明确文字，不依赖难懂图标。
- 异常原因用“少 X 件”“待处理申诉”“已锁账”这类直接表达。
- 对印尼本土员工，后续可增加印尼语短文案；当前原型先保证中文口径准确。

## 12. 财务与会计控制

结算涉及付款，必须满足以下控制：

- 每个金额必须有币种。
- 支持 `IDR`、`CNY`、`USD`，默认 `IDR`。
- 不同币种不直接合并汇总。
- 对账单生成后保存结算资料快照。
- 对账单进入预付款批次后锁账。
- 锁账后不允许直接改金额。
- 所有人工修改必须记录操作人、时间、原因、修改前后值。
- 已付款批次不反改，只能通过后续调整处理。
- 质检事实、业务确认、财务付款三层分离。

## 13. 页面体验要求

考虑业务人员容易漏看、少填、怕复杂，以及部分工厂现场人员综合素养较弱，页面必须降低理解成本。

平台端：

- 新建对账单按步骤走，不把所有字段堆在一个页面。
- 默认带出工厂建议周期和默认币种。
- 反查结果必须把“未纳入原因”直接显示出来。
- 未完成生产单置灰并标记“不纳入本期对账”，而不是隐藏。
- 关键金额修改前显示影响范围。
- 生成前显示“本期纳入 X 张生产单，排除 Y 张，扣款 Z 项”的确认摘要。

工厂端 PDA：

- 少字段。
- 大数字。
- 大按钮。
- 明确状态。
- 不展示复杂核算过程，只展示来源和结论。

## 14. 状态与异常

生产单状态：

- 已完成，可纳入
- 未完成，不纳入

对账单状态：

- 草稿
- 待工厂确认
- 工厂已确认
- 工厂申诉中
- 平台处理中
- 可入预付款批次
- 已入预付款批次
- 已付款
- 已关闭

常见异常：

- 同一流水已进入其他对账单。
- 同一生产单未完整。
- 生产单差额不为 0。
- 来源质检单存在未补齐瑕疵原因。
- 金额缺币种。
- 收款账户缺失。
- 对账单已锁账，不允许修改。

## 15. 推荐落地方案

采用“透明反查 + 完成门槛 + 对账确认”方案。

理由：

- 满足业务当前“先跑起来”的要求。
- 不把原型项目改造成复杂财务后端。
- 保留人工判断扣款的现实需求。
- 明确了质检事实和财务生效的边界。
- 能让业务人员看懂未纳入原因。
- 满足财务锁账和追溯要求。

不采用的方案：

- 不继续让质检记录直接生成扣款，因为会导致质检、对账单、扣款分析三套金额口径不一致。
- 不做全自动财务引擎，因为当前业务仍需要业务人员判断瑕疵扣款和延误扣款。
- 不隐藏未完成生产单，因为业务人员需要知道系统反查到了什么、为什么没有进入本期对账。

## 16. 验收标准

产品验收：

- 创建对账单时可选择自定义时间段。
- 创建对账单时可选择按预结算流水或按生产单。
- 按生产单时，全部反查生产单可见。
- 未完成生产单不可进入本期对账。
- 未完成生产单显示差额。
- 每张生产单展示完整数量明细和交出记录维度明细。
- 后道质检反工接收对象会影响完成判断。
- 后道工厂接收返工计入数量，但金额进入反扣。
- 质检记录不再提供扣款编辑。
- 扣款分析只统计对账单确认数据。
- PDA 只展示、确认、申诉，不做核算编辑。
- 预付款批次只消费已确认可付款对账单。

财务验收：

- 金额均有币种。
- 默认币种为 `IDR`。
- 支持 `IDR`、`CNY`、`USD`。
- 锁账后金额不可直接修改。
- 已付款批次不被反改。
- 来源、确认、付款链路可追溯。

## 17. 后续实现边界

实现时应优先在现有原型结构内最小改动：

- 先调整对账单生成体验和 mock 数据。
- 再补生产单反查和完成判断投影。
- 再拆清质检记录与扣款数据边界。
- 再调整扣款分析来源口径。
- 最后同步 PDA 展示。

不引入真实后端、真实财务系统、复杂状态管理或新 UI 框架。
