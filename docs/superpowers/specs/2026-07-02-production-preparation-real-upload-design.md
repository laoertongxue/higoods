# 生产准备时效真实上传与下载记录设计

## 背景

当前 `生产准备时效` 页面已经具备准备台账、月度统计、商品类型确认、准备项展示、花型师分配和花型上传原型。但代码核查后确认，现有“上传完成图片”仍是通过 URL 参数 `mockCompletionUploaded=1` 改变渲染状态，不是真实读取文件、保存上传记录和回写状态。

本次调整只聚焦 `生产准备时效` 页面内的真实准备项操作，不新建独立业务模块。

参考对象：

- 图片：`/Users/laoer/Desktop/生产准备阶段时效管控.png`
- 页面：`/fcs/production/preparation-timing`
- 代码：`src/pages/production/preparation-timing.ts`
- 数据：`src/data/fcs/production-preparation-timing.ts`
- 检查：`scripts/check-production-preparation-timing.ts`

## 目标

- 创建生产准备记录后，第一步由跟单确认该商品的准备工作项。
- 跟单确认后，各团队才能对各自工作项执行操作。
- 每个工作项使用独立操作弹窗，不在详情抽屉里混合所有操作。
- 上传操作必须真实读取用户选择的文件，在原型运行态中保存文件信息和内容。
- 上传完成后按工作项更新状态和完成时间。
- 基码纸样需要额外记录每一次下载时间。
- 产出状态按产出对象分行展示：`产出对象名称 + 产出时间`。

## 非目标

- 不接真实后端接口。
- 不接真实 OSS、CDN、文件服务器。
- 不设计数据库表。
- 不引入权限系统。
- 不把生产准备流程拆成新的系统模块。

说明：本仓库是前端产品原型。这里的“真实功能”指浏览器内真实选择文件、读取文件、保存到本地运行态、可下载、可记录下载日志；不是只用 query 参数模拟状态。生产环境上线时需要后端上传接口和下载审计接口承接同一数据结构。

## 核心流程

### 1. 记录进入

预售订单达到做大货要求后生成一条生产准备记录。

记录初始状态：

- `待跟单确认工作项`
- 系统已根据商品工艺和品类推导商品准备类型。
- 系统已生成默认必做项和选填项草稿。
- 各团队不能上传工作项产物。

### 2. 跟单确认工作项

跟单打开“确认工作项”弹窗。

弹窗内容：

- 商品信息：SPU、商品名、选品、买手、跟单。
- 系统推导类型。
- 跟单确认类型。
- 必做项列表，只读展示。
- 选填项列表，可勾选。
- 人工修正原因，在确认类型不同于系统推导类型时必填。

提交后：

- 写入 `workItemsConfirmedBy`。
- 写入 `workItemsConfirmedAt`。
- 选中的工作项变为可执行。
- 记录状态进入 `进行中`。

### 3. 团队操作工作项

每个工作项有自己的操作入口和独立弹窗。

详情抽屉只用于查看完整信息，不承载提交操作。

不同工作项的操作要求：

| 工作项 | 操作内容 | 完成时间规则 |
| --- | --- | --- |
| 梭织基码纸样 | 上传 `.prj` 等纸样文件；记录每次下载 | 最后一次上传时间 |
| 毛织基码纸样 | 上传 `.prj` 等纸样文件；记录每次下载 | 最后一次上传时间 |
| 梭织齐码纸样 | 上传齐码纸样文件 | 最后一次上传时间 |
| 毛织齐码纸样 | 上传齐码纸样文件 | 最后一次上传时间 |
| 版衣制作 | 上传模特试版照或成衣照片 | 最后一次上传时间 |
| 数码印/DTF/DTG花型 | 上传花型照片和花型文件 | 最后一次上传时间 |
| 染色调色（纱线） | 上传调色照片、色卡或样品照片 | 最后一次上传时间 |
| 染色调色（面料） | 上传调色照片、色卡或样品照片 | 最后一次上传时间 |
| 辅料下单 | 记录下单时间，可上传下单凭证 | 下单时间 |

### 4. 上传记录

每次上传都要保留一条独立记录，不覆盖旧记录。

字段：

```ts
interface PreparationUploadRecord {
  uploadId: string
  recordId: string
  itemId: string
  itemType: PreparationItemType
  fileName: string
  fileType: string
  fileSize: number
  fileDataUrl: string
  uploadedBy: string
  uploadedAt: string
  note: string
}
```

规则：

- `uploadedAt` 使用真实提交时的当前时间。
- 一个工作项可以有多条上传记录。
- 工作项完成时间取该工作项最后一次有效上传时间。
- 上传记录展示在工作项卡片内，按时间倒序。

### 5. 下载记录

基码纸样必须记录每次下载。

适用工作项：

- `梭织基码纸样`
- `毛织基码纸样`

字段：

```ts
interface PreparationDownloadRecord {
  downloadId: string
  recordId: string
  itemId: string
  uploadId: string
  fileName: string
  downloadedBy: string
  downloadedAt: string
}
```

规则：

- 点击下载按钮时，先写入下载记录，再触发浏览器下载。
- 每次点击都生成一条下载记录。
- 详情抽屉和工作项弹窗都显示下载记录。
- 非基码纸样可以下载文件，但不强制展示下载审计。

### 6. 状态变更

工作项状态：

- `待跟单确认`：记录已进入，但跟单未确认工作项。
- `待开始`：跟单已确认，该项可执行。
- `进行中`：已分配或已上传部分资料，但未完成。
- `待确认`：该项需要买手或跟单确认。
- `已完成`：该项完成时间已生成。
- `已超时`：超过计划完成时间且未完成。
- `无需`：选填项未被跟单选择。

最小原型规则：

- 普通上传类工作项提交成功后直接进入 `已完成`。
- 花型上传后进入 `待确认`，买手确认通过后进入 `已完成`。
- 辅料下单提交下单时间后进入 `已完成`。
- 未确认工作项时，各工作项操作按钮置灰。

### 7. 产出展示

产出不再只显示“统一生成时间”。

每个产出对象独立展示：

| 展示字段 | 说明 |
| --- | --- |
| 产出对象名称 | 例如正式技术包、生产单、印花单、染色单、辅料采购单 |
| 产出对象编号 | 已生成时显示正式编号；未生成时显示预计编号 |
| 产出状态 | 预计生成 / 已生成 |
| 产出时间 | 已生成时显示具体时间；未生成时显示 `-` |

产出生成规则：

- 所有已选择工作项完成后，才显示正式产出。
- 产出时间取所有已选择工作项完成时间中的最大值。
- 每个产出对象都保存自己的 `outputGeneratedAt`。

```ts
interface ProductionPreparationOutput {
  outputType: PreparationOutputType
  outputNo: string
  outputHref: string
  outputStatus: PreparationOutputStatus
  outputGeneratedAt: string
}
```

## 页面调整

### 准备台账

行操作保留：

- 查看详情
- 确认工作项
- 操作当前卡点
- 查看生产单

不再从列表直接进入“完整详情里的上传区域”。

### 工作项操作弹窗

根据工作项类型显示不同表单：

- 文件上传类：文件选择、说明、提交。
- 花型类：图片文件、花型源文件、说明、提交。
- 辅料下单：下单时间、凭证文件、说明、提交。
- 基码纸样：上传区、历史上传、下载按钮、下载记录。

### 详情抽屉

详情抽屉只读展示：

- 来源信息。
- 商品类型确认。
- 准备项确认结果。
- 工作项明细。
- 上传记录。
- 基码纸样下载记录。
- 产出对象列表。
- 操作日志。

## 本地持久化

因为没有后端，本次使用 `localStorage` 保存运行态。

保存内容：

- 工作项确认结果。
- 上传记录。
- 下载记录。
- 工作项状态变更。
- 产出生成时间。

存储键：

```ts
const PREPARATION_RUNTIME_STORAGE_KEY = 'higood.production-preparation.runtime.v1'
```

重置浏览器本地存储后，页面回到 mock 初始状态。

## 验收

- `npm run check:production-preparation-timing` 通过。
- `npm run build` 通过。
- 页面能打开 `/fcs/production/preparation-timing`。
- 未确认工作项的记录不能上传产物。
- 确认工作项后，工作项按钮可操作。
- 上传文件后，页面显示真实文件名、文件大小、上传人、上传时间。
- 重新打开页面后，上传记录仍存在。
- 基码纸样点击下载后，新增下载记录。
- 多次上传和多次下载都逐条保留。
- 产出对象按行展示，并显示每个对象的产出时间。
