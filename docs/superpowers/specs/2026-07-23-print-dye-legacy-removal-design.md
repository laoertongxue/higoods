# 印染旧链路彻底清理设计

## 已确认目标

删除已取消的“印花需求单、染色需求单”对象、编号、字段、投影、页面口径和兼容测试；不保留兼容层。印花、染色实际加工单只允许来自生产单、备货或裁片补料。生产需求单仍保留；BOM 的“是否需印花 / 是否需染色”仍为工艺属性，不是单据。

## 唯一事实源

- 生产单来源：冻结技术包 / BOM → `ProcessWorkOrder`。
- 备货来源：手动创建 `ProcessWorkOrder`。
- 补料来源：补料确认 → `ProcessWorkOrder`。
- 仓库、PDA、交出、后道只关联加工单、任务、生产单、BOM 和补料记录；不得存储 `sourceDemand*` 或需求编号。

## 清理边界

1. 删除准备页适配器的 `PrepRequirement*`、需求编号、DEMAND artifact fallback 及导出 API；准备页只从实际加工单投影。
2. 删除仓库领域的 `sourceDemandId/sourceDemandNo` 和印染需求号种子；不得用旧字段承载任务号。
3. 生产单进度与 PFOS 文案改为印花加工单 / 染色加工单、加工计划数量，使用真实加工单来源，不展示旧需求节点。
4. 删除生产产物中针对印花、染色的需求产物编号和特判；保留通用生产需求和非印染工艺的既有语义。
5. 成衣流程每一步必须有至少一个 SKU 数量大于 0；后道交出必须按可回滚边界写入，失败不得留下半写状态。
6. 浏览器测试使用当前 operation slug，真实执行成衣仓出库 → 辅助收货 → 加工完工 → 待交出 → 后道收货；不允许仅验证静态文案。

## 验收

- 扩大关键词扫描不匹配旧印染需求对象、编号、字段或投影 API。
- 对生产需求、BOM 工艺属性、WLS 物料所需量采用白名单说明，不被误删。
- 零 SKU 动作被拒绝且不改变状态。
- 后道写入故障可恢复至动作前状态。
- 现行特殊工艺仓联动浏览器用例和新的成衣 E2E 用例均通过。
