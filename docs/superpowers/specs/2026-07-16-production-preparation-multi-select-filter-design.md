# 生产准备时效多选筛选与准备项联动设计

## 1. 背景

生产准备时效当前包含三个列表视图：准备台账、月度统计、明细统计。现有筛选以原生单选下拉为主，准备台账展示买手、责任人、是否超时等条件，但无法同时筛选多个跟单、状态、准备项或团队，也没有按准备项是否具备开始条件区分进度。

当前代码已经具备以下事实：

- 每条生产准备记录包含跟单、记录状态和多个准备项。
- 每个准备项包含准备项类型、责任团队、依赖项、状态和完成凭证。
- 项目已有 `renderMultiSelectFilter` 下拉复选框组件。
- 项目已启用标准列表页强制门禁，被调整的列表页必须使用标准列表页、标准表格和标准分页组件。

本设计只调整生产准备时效三个列表视图的筛选和列表治理，不改变准备工作项确认、上传、下载、辅料下单、产出生成和月度完成数量口径。

## 2. 目标

1. 删除准备台账、月度统计和明细统计中的“责任人”“是否超时”筛选。
2. 将筛选中的“买手”改为“跟单”。
3. 跟单、记录状态、准备项、责任团队全部支持多选。
4. 准备台账新增“准备项进度”多选筛选。
5. 准备项与责任团队双向联动，避免组合出业务上不存在的条件。
6. 保持月度统计和明细统计的完成数量口径，不在统计页增加准备项进度筛选。
7. 筛选、联动、分页和列表交互响应不超过 200ms。
8. 准备台账、月度统计和明细统计同步满足标准列表页治理门禁。

## 3. 非目标

- 不增加新的准备项状态。
- 不修改准备项依赖关系。
- 不修改超时计算，只移除本次页面中的超时筛选入口。
- 不删除买手、责任人、超时等底层数据字段。
- 不修改月度统计“按实际完成时间所在月份统计”的口径。
- 不引入后端、权限系统、状态管理框架或新的 UI 框架。

## 4. 页面范围

### 4.1 准备台账

筛选顺序固定为：

1. 日期
2. 跟单
3. 记录状态
4. 准备项
5. 准备项进度
6. 责任团队
7. 关键词
8. 筛选
9. 重置

跟单、记录状态、准备项、准备项进度、责任团队使用下拉复选框。日期仍是一个起止日期区间，关键词仍是单个文本输入。

### 4.2 月度统计与明细统计

筛选顺序固定为：

1. 月份或日期口径
2. 跟单
3. 记录状态
4. 准备项
5. 责任团队
6. 关键词及现有统计动作

跟单、记录状态、准备项、责任团队使用下拉复选框。统计页不增加“准备项进度”，因为月度统计和明细统计的数据源只包含具备实际完成事实的准备项。

## 5. 多选交互

### 5.1 控件形式

复用现有 `renderMultiSelectFilter`：

- 默认只展示字段名和已选数量，例如“跟单（2）”。
- 点击后打开复选框下拉层。
- 未选择任何选项表示不限制该维度。
- 用户点击“筛选”后才更新 URL 和列表。
- 勾选过程只局部更新准备项与责任团队两个联动下拉，不触发整页重绘。

### 5.2 组合规则

- 同一筛选条件内的多个值按“或”匹配。
- 不同筛选条件之间按“且”匹配。
- 点击筛选后页码重置为第 1 页。
- 分页、刷新和浏览器前进后退必须保留全部多选值。

示例：跟单选择 Maya、Raka，准备项选择毛织基码纸样、毛织齐码纸样，责任团队选择毛织团队，表示：

- 跟单是 Maya 或 Raka；
- 同一个准备项是毛织基码纸样或毛织齐码纸样；
- 该准备项的责任团队是毛织团队。

## 6. 准备项与责任团队联动

### 6.1 固定映射

| 准备项 | 责任团队 |
| --- | --- |
| 梭织基码纸样 | 版师团队 |
| 毛织基码纸样 | 毛织团队 |
| 版衣制作 | 车板团队 |
| 梭织齐码纸样 | 版师团队 |
| 毛织齐码纸样 | 毛织团队 |
| 数码印/DTF/DTG花型 | 花型团队 |
| 确认染色要求（纱线） | 跟单角色 |
| 染色调色（纱线） | 染色团队 |
| 确认染色要求（面料） | 跟单角色 |
| 染色调色（面料） | 染色团队 |
| 辅料下单 | 采购团队 |

该映射作为生产准备筛选的唯一口径，准备台账和统计页共用，不从当前筛选结果临时推断。

### 6.2 双向联动规则

- 已选择准备项时，责任团队候选项只保留这些准备项对应团队的并集。
- 已选择责任团队时，准备项候选项只保留这些团队负责准备项的并集。
- 多选时采用并集，避免只能筛选单一团队。
- 已选值必须保持可见，不能因为候选项变化而形成隐藏筛选条件。
- 正常界面操作应阻止产生不兼容组合；通过旧链接或手工修改 URL 形成不兼容组合时，保留选择并展示标准无结果状态。

## 7. 准备项进度口径

准备项进度不是新增状态，而是根据确认事实、依赖关系和完成凭证派生的筛选维度。

### 7.1 不满足开始条件

满足任一条件即归入：

- 所属生产准备记录尚未完成跟单工作项确认；
- 准备项存在前置依赖，且任一前置准备项没有有效完成凭证。

### 7.2 未开始

同时满足：

- 跟单已经确认工作项；
- 所有前置准备项均具备有效完成凭证；
- 当前准备项尚未具备有效完成凭证。

该档覆盖底层状态为待开始、进行中、待确认、已超时等尚未完成的准备项，避免形成筛选盲区。

### 7.3 已完成

准备项状态为已完成、存在实际完成时间，并具备对应业务凭证：

- 辅料下单：至少存在一个面辅料采购单号，完成时间取最后一张采购单的下单时间；
- 其他准备项：至少存在一条有效上传记录，包含文件、上传人和上传时间。

### 7.4 同一准备项匹配

准备项、责任团队、准备项进度三个条件必须由同一个准备项同时满足。

不得出现以下误命中：同一记录中“毛织基码纸样”满足准备项条件，而另一个“辅料下单”满足采购团队条件，系统却把该记录判定为命中。

## 8. URL 与筛选模型

### 8.1 URL 参数

沿用现有参数名并允许重复：

- `merchandiserName`
- `recordStatus`
- `itemType`
- `ownerTeam`
- `itemProgress`，仅准备台账

示例：

```text
?merchandiserName=Maya&merchandiserName=Raka&itemType=毛织基码纸样&itemType=毛织齐码纸样&ownerTeam=毛织团队
```

单值旧链接天然兼容，因为单个重复参数等同于只包含一个值的数组。

准备台账和统计页不再读取 `buyerName`、`ownerName`、`overdueOnly` 作为筛选条件；底层字段保留。

### 8.2 数据模型

筛选模型使用数组表达多选：

- `merchandiserNames`
- `recordStatuses`
- `itemTypes`
- `ownerTeams`
- `itemProgresses`，仅准备台账

日期和关键词继续使用单值。

### 8.3 导航收集

通用字段导航需要支持同名复选框：

- 勾选的同名复选框使用 `URLSearchParams.append` 依次写入；
- 普通输入框、日期和单选字段仍保持单值；
- 未勾选复选框不写入 URL；
- 筛选提交自动写入 `page=1`。

## 9. 过滤数据流

### 9.1 记录级条件

日期、跟单、记录状态、关键词在记录层匹配：

- 多个跟单按或匹配；
- 多个记录状态按或匹配；
- 日期与关键词继续按现有口径匹配。

### 9.2 准备项级条件

准备项、责任团队、准备项进度在准备项层合并判断：

1. 取记录内已选择且状态不是“无需”的准备项；
2. 对每个准备项同时判断准备项类型、责任团队和派生进度；
3. 只要存在一个准备项同时满足所有已启用的准备项级条件，该记录即通过准备项级筛选；
4. 没有启用任何准备项级条件时，不限制记录。

统计页沿用相同的准备项与团队匹配，但统计明细仍只统计实际完成时间位于目标月份的完成准备项。

## 10. 标准列表页治理

本次会调整既有列表页筛选，必须同步完成以下治理：

- 页面源文件声明 `// @page-pattern: list`；
- 准备台账、月度统计和明细统计使用 `renderStandardListPage`；
- 表格使用 `renderStandardListTable`；
- 分页使用 `renderTablePagination`；
- 操作列固定右侧；
- 支持列显示、列顺序、普通列冻结和数据排序；
- 每页条数按路由保存，页码和排序返回稳定默认值；
- 1366×768 下保持完整可用，1280×720 下页面主体不横向溢出，宽表只在表格容器内滚动。

标准化只调整列表骨架和通用能力，不重写生产准备确认、上传、详情抽屉和产出逻辑。

## 11. 性能与交互

- 打开或关闭多选下拉只操作当前下拉 DOM。
- 勾选准备项或责任团队只更新两个联动下拉的候选项和已选数量。
- 关键词输入不触发整页重绘。
- 点击筛选后才进行路由导航和列表重渲染。
- 所有按钮和复选框的可见响应必须小于 200ms。
- 图标初始化仅作用于新插入或局部更新区域。

## 12. Mock 数据覆盖

现有 Mock 数据需要逐条复核，并确保至少覆盖：

1. 跟单未确认工作项，准备项进度为不满足开始条件；
2. 前置准备项未完成，后续准备项为不满足开始条件；
3. 前置条件已满足但尚未上传，准备项进度为未开始；
4. 底层状态为进行中、待确认、已超时，但派生进度仍为未开始；
5. 普通准备项具备完整上传凭证，派生进度为已完成；
6. 辅料下单具备多个采购单号和最后下单时间，派生进度为已完成；
7. 一条记录包含多个不同团队准备项，用于验证同一准备项匹配；
8. 月度统计和明细统计存在多个跟单、准备项和责任团队的完成记录，用于验证多选统计。

如果现有数据已经覆盖，不重复增加；缺少的场景只补最少必要 Mock 数据。

## 13. 验证与验收

### 13.1 自动检查

- 跟单替代买手；责任人和是否超时不再出现；
- 准备台账存在三档准备项进度；统计页不存在准备项进度；
- 五类台账选项筛选和四类统计选项筛选均为多选；
- 同组多选按或、不同组按且；
- 准备项、团队、进度必须在同一准备项上匹配；
- 准备项与责任团队双向联动候选项正确；
- 普通完成凭证和辅料采购单完成凭证均能正确分类；
- 重复 URL 参数在筛选、分页、刷新后保留；
- 月度完成数量口径不因筛选改造发生变化；
- Mock 数据覆盖第 12 节全部场景。

### 13.2 项目门禁

至少运行：

```bash
npm run check:production-preparation-timing
npm run check:list-page-governance
npm run check:prototype-design-governance -- --all
npm run build
```

### 13.3 浏览器验收

- 1366×768 和 1280×720 两个桌面视口；
- 准备台账多选、双向联动、三档进度、筛选、重置和分页；
- 月度统计与明细统计多选、联动、分页和导出入口；
- 旧单值链接兼容；
- 无结果状态可见；
- 筛选区、表格、操作列无重叠或页面级横向溢出；
- 点击响应小于 200ms。

## 14. 最终结论

本次改造把生产准备时效筛选从单值、记录级粗筛选调整为多值、准备项事实级筛选。准备项、责任团队和派生进度在同一个准备项上完成匹配，准备台账可以准确定位当前可处理与被前置条件阻塞的任务；统计页同步多选与联动，但保持完成数量统计口径不变。
