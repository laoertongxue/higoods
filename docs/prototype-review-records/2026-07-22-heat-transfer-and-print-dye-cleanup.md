## 6. 最终结论

结论：通过

说明：

- 烫画与直喷的菜单、动态路由、标题、加工单业务类型、加工单号和筛选状态已隔离。
- 两类加工单均保留裁片部位与成衣演示记录。
- 成衣链路已按“成衣仓通过 Web / PDA 逐 SKU 出库 → 辅助工艺待收货 → 逐 SKU 收货入待加工仓 → 加工 → 逐 SKU 待交出 → 我方后道逐 SKU 收货”实现；未出库和仅出库未收货均不会形成待加工库存。
- 成衣仓出库、辅助工艺收货、完工和后道收货均走统一动作或领域服务；仓联动失败不会先改工单状态，已收货状态不能重复执行。
- 待加工领用与待交出确认使用独立进程的 PDA 行为契约验证真实页面事件、明确卡片身份、状态推进、库存扣减、交接记录、重复操作拒绝和登录审计；测试运行时统一重置会话与本地存储，避免前序场景污染。
- 多 SKU 行为契约验证点击 SKU A 只处理 A、SKU B 继续留在操作列表，最后一个 SKU 完成后才推进整单；同时通过真实 PDA dispatch 验证未登录和重复操作会显示用户可见的中文 Toast。
- 本次样例将计划 500 件的一个 SKU 从成衣仓实出 498 件、辅助工艺实收 496 件；收货前 PDA 默认实收为 498，收货后固定显示应收 498、已收 496、差异 2、可加工 496，不再开放重复实收；加工环节对指定 SKU 报废 1 件、后道环节对另一个指定 SKU 少收 1 件，差异均落到实际 SKU。
- 契约使用真实 PDA 登录会话验证：无会话拒绝、辅助工艺厂账号拒绝成衣仓出库、现有中央成衣厂管理员可从任务列表进入并成功出库、对应辅助工艺厂账号可继续收货与加工；同时覆盖重复操作拒绝和烫画 / 直喷同 SKU 草稿隔离。
- 加工执行逐 SKU 保存已收、完工、报废和货损；后道使用用户提交的逐 SKU 实收映射生成一条与生产单、烫画加工单关联的既有后道任务。
- 成衣链路不生成菲票、不展示部位、不回裁床；裁片链路仍按片流转并保留已有菲票事实。
- 技术包维护人员在 BOM 中选择“成衣”时，系统固定为件、用量 1、损耗 0，并要求关联 SKU；同一物料的工艺路线只允许先染色、后印花。成衣不会被带入裁片、纸样或菲票链路。
- 管理端使用独立的“烫画加工单”和“直喷加工单”入口；各自的路由、标题、加工单业务类型与列表偏好隔离。裁片详情保留部位和菲票事实，成衣详情仅保留 SKU、件数、成衣仓来源和我方后道工厂去向。
- 成衣仓人员在 Web / PDA 逐 SKU 确认实出；辅助工艺操作员仅可在所属工厂收货、加工并交出；我方后道工厂收货员按 SKU 实收。系统自动计算应收、已收、可加工和差异，不要求现场人员心算；超量、错厂、未登录与重复提交均阻断并显示中文处理提示。
- PDA 只呈现当前 SKU、件数、当前动作和结果；成衣不显示裁片部位或菲票，不需要填写虚拟菲票。裁片仍要求绑定真实菲票，两个对象不能混用。
- 在 1366×768 标准分辨率与 1280×720 最低分辨率下，宽表仅在表格容器横向滚动，操作列固定可见；PDA 的主动作保持可见，弹窗、抽屉和详情不会覆盖确认入口。
- 无例外：本次不保留设计、页面、列表治理或交互性能例外；未修改治理脚本或历史基线以绕过检查。
