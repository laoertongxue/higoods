## 3. 业务边界

- 裁片单关闭或完成在放行矩阵中都映射为“冻结最后有效贡献”，但原裁片单生命周期原因继续保留；本任务不新增“完成”按钮或另一套终态。
- 重新打开只恢复“持续更新”资格，不自动吸收关闭期间已经到达的迟到铺布事实。
- 迟到铺布不是有效数量变化，因此不生成有效矩阵版本；后续必须重新提交合法新事件，或走冲销/修正确认。
- 原型现有裁片单与 PO14671 矩阵属于不同 Mock 数据域。本任务在裁片单页面增加由同一放行仓储状态生成的 `CUT14671-A/B` 联动演示行，不复制或重算矩阵。
- 本地多周期记录 ID 使用历史最大合法周期加一，并追加随机 nonce，例如 `close:<cutOrderId>:cycle-10000-<nonce>`；乱序、重复、非法或 `cycle-9999` 历史不会造成覆盖。旧版固定 ID 继续可读。
- 同记录 ID 且内容一致按幂等成功处理；同记录 ID 但内容不同明确返回冲突，不覆盖历史。当前关闭状态由关闭/重开事件时间线的最新业务事件决定，不再依赖数组长度或插入顺序；测试清理也按记录 ID 精确删除。
- 操作令牌仅作为页面内 in-flight 防重锁，事务结束后统一释放；失败会局部生成新令牌供立即重试，业务 exactly-once 由 operationKey 负责，不依赖随机令牌。
- 页面先幂等写入本次关闭/重开账，再更新裁片单 Web 阶段；阶段更新异常时回滚刚写入的账，不产生放行事件。只有两者成功后才用该业务记录 ID 写放行事件。
- 阶段更新不仅检查异常，也检查返回记录；找不到同 ID 投影时按失败处理。PO14671 演示投影中的数量字段固定为 0，不参与齐套、目标或补料计算，矩阵数量仍只来自裁片放行事实仓储。

## 4. 例外

- 当前原型没有裁片单“完成”专属页面动作；已通过公开放行状态事件验证“已完成，数据已冻结”的原因展示，未扩展原生命周期状态机。
- active spreading 的演示状态来自放行仓储关联铺布来源；真实系统应由铺布单状态账实时提供。

## 5. 最终结论

结论：通过。关闭/完成冻结而非删除、进行中铺布阻断、重开恢复资格、迟到数据不计入及历史追溯均有中文页面表达和自动化覆盖。

自动化复核：真实 SPA 用例只在首次进入应用时加载页面，后续统一使用应用内导航，覆盖 `V1 冻结 → V2 重开 → V3 关闭 → V4 再重开 → V5 再关闭`；每轮同步检查阶段投影、矩阵页面的“已冻结，不再更新”或“持续更新”、本轮原因和来源，以及 M/L/XL 仍为 200/350/500。同时验证两轮关闭/重开记录 ID 不同、四条历史均保留、事件重放不增版本、第一次真实关闭后收到的迟到铺布不改变矩阵数量，并定向覆盖缺失阶段投影、放行拒绝/抛错时完整回滚、乱序/超大周期/冲突写入、快速双击幂等及局部刷新低于 200ms。
