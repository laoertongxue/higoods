### 1.1 上下游与责任边界

- 上游：生产单、工艺路线、车缝分配工作台、连续工序任务、工厂主数据。
- 下游：PDA 接单 / 执行 / 交出、接收方确认实收、生产进度与异常责任复核。
- 分配事实：独立车缝只进入车缝分配工作台；含车缝连续任务只进入连续工序任务页面。
- 接单事实：直接派单按业务分配时间自动接单；竞价定标后由中标工厂明确确认接单，确认时才启动时效。
- 数量事实：工厂交出不等于回货完成，只有接收方确认实收才计入履约；超量实收正常计入，履约比例允许超过 100%。
- 主工厂事实：一个生产单只有一个主工厂；只有一个有效车缝工厂时系统自动选择，多工厂时要求管理人员显式选择。

## 5. 主要问题与处理

| 问题 | 标签 | 影响角色 | 处理方式 | 是否仍有风险 |
| --- | --- | --- | --- | --- |
| 独立车缝和连续工序任务入口边界容易混淆 | `协作断裂` | 生产计划员 / 跟单 | 车缝工作台排除连续任务，连续任务页单独承接分配动作。 | 否 |
| 业务分配时间可回填但可能误填未来时间 | `点错风险` | 生产计划员 / 跟单 | 统一以操作端本地墙钟校验，未来时间直接阻断且不产生部分状态。 | 否 |
| 竞价定标可能被误认为已经接单 | `状态抽象` | 工厂主管 | 定标后保持“待确认接单”，由工厂确认时创建履约快照并启动滚动 24 小时时钟。 | 否 |
| 工厂交出数量被误当作回货完成数量 | `协作断裂` | 跟单、工厂、接收方 | 只累计接收方确认实收；同时展示累计交出与累计已确认实收。 | 否 |
| 接收方晚确认导致工厂看似逾期 | `缺主管兜底` | 平台主管 | 仍按真实确认时间计算，同时展示接收延迟记录并提供“主管复核责任”。 | 否 |
| 改派后旧实收和新任务时钟可能混用 | `追溯不足` | 跟单、平台主管 | 旧快照失效但保留历史；新任务按剩余未实收数量创建独立快照，旧任务后续实收不污染新快照。 | 否 |
| 多家车缝工厂时主工厂归属不明确 | `选不对` | 生产计划员 / 跟单 | 单厂自动选择；多厂要求显式选择且全程保持唯一主工厂。 | 否 |
| 独立车缝部分 SKU / 部分数量曾只生成演示记录 | `协作断裂` | 生产计划员 / 跟单、工厂 | 复用运行时明细拆分模型，把原范围原子分区为本次分配子任务和剩余子任务；源任务停用，下游等待全部分区，成功记录必须关联真实 runtime task / tender。 | 否 |
| 多 SKU 弹窗无法取消单行分配 | `点错风险` | 生产计划员 / 跟单 | 每个 SKU 提供独立勾选，默认全选但允许取消；未选行数量禁用且提交忽略，至少保留一个 SKU 才能提交。 | 否 |
| 改派后旧任务履约历史无法继续查看 | `追溯不足` | 跟单、平台主管 | 当前快照与历史最新快照分离读取；旧 taskId 后续实收继续推进 inactive 历史快照，且可按该 snapshotId 复核接收延迟责任。 | 否 |
| PDA 拒单曾直接修改旧任务事实 | `协作断裂` | 工厂主管、工厂操作员 | PDA 列表与详情统一调用运行时拒单入口，校验待接单状态和工厂身份，写入中文拒单审计；重复 / 错厂拒单无状态。 | 否 |
| 已拒单任务仍可能出现在已中标或执行入口 | `点错风险` | 工厂主管、工厂操作员 | 已中标集合统一排除 `REJECTED`；详情深链保留“已拒单”追溯，但明确阻断确认接单与去执行。 | 否 |
| 弹窗停留时间可能被误当作实际分配时间 | `追溯不足` | 生产计划员 / 跟单 | 业务时间仍可回填；直接派单、竞价和改派均在点击提交时重新读取操作端墙钟，审计和 assignmentOperatedAt 不复用打开弹窗时间。 | 否 |
| 工厂拒单后任务无法重新定标 | `协作断裂` | 跟单、工厂主管 | 保留拒单审计与 tender 历史，释放当前中标归属并回到可定标状态；新工厂定标后重新进入待接单并按新确认时间启动快照。 | 否 |
| 直接派单拒单被错误送回竞价 | `状态抽象` | 跟单、工厂主管 | 拒单按原分配方式分流：竞价任务保留 tender 及原业务 / 操作分配时间并回到可定标；直接派单任务回到未分配并释放当前工厂、价格和分配时间，可再次直接派单。 | 否 |
| 分批派给新工厂可能误改生产单主工厂 | `选不对` | 生产计划员 / 跟单 | 按生产单独立判断：已有有效主工厂默认保留；无主工厂且唯一候选自动确认；多候选要求逐生产单显式选择，跨生产单批量不共用一个选择。 | 否 |
| 弱网下接收事实与仓储联动可能不同步 | `协作断裂` | 接收方仓管、主管 | 已成功的实收事实不回滚，页面提示仓储联动待同步并保留主管处理入口。 | 否 |
| 改派后旧分配虽保留数据但管理端无法搜索和进入详情 | `追溯不足` | 跟单、平台主管 | 进度看板新增“已改派（历史）”只读行，可按旧 taskId、旧工厂和新 taskId 搜索；详情展示旧工厂、分配量、累计交出 / 实收、30% / 70% / 100% 节点及改派关系。历史行不计入执行 KPI，也不提供执行动作；旧 taskId 后续实收继续刷新历史投影。 | 否 |
| 开发热重载后只读桥因函数引用变化中断页面数据 | `协作断裂` | 管理端、PDA 使用者 | 稳定 bridge / registry 在同模块热重载时替换为最新只读 resolver / reader；未安装仍安全返回空，任务仓和交出仓保持单一事实源。 | 否 |
| PDA 深链 taskId 含引号或方括号时聚焦选择器可能报错 | `选不对` | 工厂操作员 | 聚焦前统一执行 CSS selector 值转义；普通任务号保持不变，特殊字符不再破坏查询选择器。 | 否 |
| 实收事实版本时间缺少明确写入契约 | `追溯不足` | 接收方仓管、平台主管 | 明确 `lifecycleUpdatedAt` 是交出记录生命周期版本时间；现有实收回写 mutation 必须随接收确认时间同步推进，不额外虚构作废流程。 | 否 |

## 6. Mock 与边界验收

- 独立车缝：30%、70%、100% 三节点均按时。
- 车缝到包装：30% 逾期、70% 追上、最终按时。
- 裁片到包装：30% 按时、70% 逾期、最终逾期完成。
- 最终截止后未完成仍显示逾期待达标。
- 截止前交出、截止后确认形成接收方确认延迟。
- 101 件的 30% / 70% / 100% 目标为 31 / 71 / 101 件。
- 超量实收正常完成，履约比例可以超过 100%。
- 部分实收后改派，新旧快照、剩余数量和后续实收互相隔离。
- 单车缝工厂自动成为主工厂，多车缝工厂必须显式选择。
- 独立车缝支持部分 SKU、部分数量连续多次分配；每次分配量、剩余量及总 scope 守恒，下游依赖全部分区任务。
- 独立车缝直接派单与竞价均支持业务分配时间回填和 30% / 70% / 100% 节点预览；未来时间在运行时写入前阻断。
- 改派后旧任务新增实收只更新旧历史投影，新任务投影不变；旧历史快照可保存接收延迟责任复核。
- PDA 列表与详情拒单均落运行时 `REJECTED`，重进后保持“已拒单”且不再展示接单 / 拒单动作。

- `check:sewing-delivery-sla-final-fixes`：数量分区总量守恒、连续二次部分分配、真实 direct / tender、快照分母、未来时间阻断、统一拒单身份与重复校验。

- 对抗式补审进一步封堵同一含车缝 `runtimeTaskId` 被普通直派或竞价入口覆盖：已有工厂、已接单或生效快照时必须走改派；不同实际分配结果使用不同 `runtimeTaskId`，旧任务迟到实收只更新旧历史，不回减新任务分配量。
- 履约事实按查询时点截断：未来交出与未来实收不提前展示；跨交出单重复 `recordId` 只采用观察时点内最新有效版本；接收延迟归责数量同时受该记录截止前交出量和节点缺口约束。
- 履约快照编号同时绑定运行时任务和分配结果，并在保存时阻断跨任务 ID 冲突；单条与批量履约视图共用历史资格判断，已停用但保留历史快照的旧任务仍可查询。
- 复审补充：投影同时校验交出时间不得晚于查询时点或实收时间；交出原始记录先跨交出单按记录身份选择最新版本，再判断任务归属和确认状态，因此后续作废、退回待确认、重新确认及跨任务移动都能正确撤销或替换旧事实。
- 确定性补审：相同版本时间按生命周期等级和固定字段规范签名选择，结果不受 registry 遍历顺序影响；缺少交出时间的事实直接判无效，不扩大历史兼容口径。
- 历史时点补审：交出记录按生命周期更新时间、履约快照按接单时间先截断到查询时点；未来作废、迁移、实收或改派不会提前覆盖历史视图。连续含车缝竞价拒单释放后允许同 tender 调整重开，不同 tender 仍被原子阻断。
