## 5. 主要问题与处理

| 问题 | 标签 | 影响角色 | 处理方式 | 是否仍有风险 |
| --- | --- | --- | --- | --- |
| 特殊工艺动作注册与页面动作不一致 | `协作断裂` | Web / PDA 操作员 | 统一为 4 个动作码并清理旧动作码 | 否 |
| 特殊工艺 PDA 仍有差异/重交入口 | `点错风险` | PDA 操作员 | 删除特殊工艺差异/重交入口和处理分支 | 否 |
| 弹窗字段不能区分接收/完工/交出 | `算不准` | Web 操作员 | 按动作展示实收、完工、交出、回写列 | 否 |
| 自定义 SKU / 菲票弹窗事件不可达 | `协作断裂` | Web 操作员 | 主事件分发补充 SKU、菲票确认和列表字段事件 | 否 |
| SKU / 菲票 input key 使用中文清洗导致碰撞 | `算不准` | Web 操作员 | SKU 使用 `data-sku-code`，菲票使用 `data-fei-ticket-no` | 否 |
| PDA 特殊工艺仍有旧字段 `workOrderId/workOrderNo` | `协作断裂` | PDA 操作员 | 改为 `taskOrderId/taskOrderNo`，移除错误默认发起交出兜底 | 否 |
| 成衣 PDA / 仓内动作缺逐 SKU payload | `算不准` | PDA 操作员 | 确认接收、加工填报、发起交出均按 SKU 写回 | 否 |
| 特殊工艺打印路线卡仍混入旧节点 | `信息不一致` | 主管 / 操作员 | 路线卡节点归一为确认接收、加工填报、发起交出、完成加工单 | 否 |
| 专项检查脚本漏检平台和打印卡 | `检查缺口` | 评审人员 | 补充事件分发、平台具体加工单、打印卡节点结构化断言 | 否 |
| 弹窗挂载到 `body` 导致 `#app` 事件委托不可达 | `协作断裂` | Web 操作员 | 自定义弹窗挂载到 `#app` 内，并由专项脚本断言 | 否 |
| 数量可超收、超完工、超交出 | `算不准` | Web / PDA 操作员 | 4 动作写回增加阶段上限校验，专项脚本补负例 | 否 |
| PDA 通用开工 / 完工绕过特殊工艺状态机 | `点错风险` | PDA 操作员 | 特殊工艺隐藏通用开工 / 完工按钮，删除通用开工自动确认接收副作用 | 否 |
| 成衣确认接收后缺待加工 SKU 库存 | `协作断裂` | PDA 操作员 | 确认接收联动生成成衣 SKU 待加工库存 | 否 |
| 打印路线卡可能输出多条重复历史节点 | `信息不一致` | 主管 / 操作员 | 打印卡聚合为严格 4 个动作节点 | 否 |
| 确认接收、加工填报、发起交出弹窗确认按钮点击后无反应 | `协作断裂` | Web 操作员 | 修复详情页事件处理顺序，SKU / 菲票确认按钮绕过 Web action 早退并进入提交处理；专项脚本补可达性断言 | 否 |

## 6. 最终结论

结论：通过

说明：

- 本次只收口特殊工艺加工单简化相关 Web/PDA 原型链路，不调整后道、裁床等其他模块的差异处理能力。
- `src/data/fcs/cutting/special-craft-fei-ticket-flow.ts` 中仍保留裁片菲票回仓差异 / 异议底层模型，作为裁床回仓历史旁路能力；专项检查已约束这些字段不得出现在特殊工艺加工单页面、PDA 主入口和打印路线卡 4 操作主链路中。
- 第二轮对抗审查新增运行时负例：确认接收超计划、加工填报超已接收、发起交出超已完工未交出、未交出前完结均必须失败。
- 第三轮问题核查确认：自定义 SKU / 菲票弹窗挂载已在 `#app` 内，但详情页事件处理在非 `[data-special-craft-web-action]` 点击时提前返回，导致弹窗内确认按钮不可达；已修复并纳入专项脚本防回归。
