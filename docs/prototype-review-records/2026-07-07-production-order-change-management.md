## 3. 自查结论

| 检查项 | 结论 | 说明 |
| --- | --- | --- |
| 角色匹配 | 通过 | 页面定位为 FCS 管理端，服务跟单、买手、工艺、仓管、裁剪、印花、染色、车缝、结算、计划履约等跨角色协同；管理端负责变更意图录入、影响推导和审核记录；主管端负责高风险整单暂停、跨角色冲突和异常兜底确认；下游执行端只接收相关单据、新旧口径和当前处理动作。 |
| 任务清晰度 | 通过 | 业务人员不需要先判断变更类型，而是先录入变更意图和现场事实，系统再反推版本关系变更、生产单层补丁、单据处理和成本 / 时效影响。 |
| 信息架构与导航 | 通过 | 页面保留在 `/fcs/production/changes`，以生产单变更管理承接待处理、版本关系、生产补丁、单据处理、料工费差异、时效影响和处理记录。 |
| 页面模式 | 通过 | 管理端可使用表格、抽屉、筛选和日志表达全链路判断；现场端感知被收敛为相关单据的新旧口径和当前处理动作。 |
| 信息负荷 | 通过 | 复杂判断放在管理端，不把完整技术包版本模型、影响矩阵和费用推导暴露给一线执行人员。 |
| 文案 | 通过 | 页面使用中文业务词表达生产单变更、版本关系变更、生产单层补丁、单据处理、成本影响和时效影响，避免让现场人员理解系统内部状态码。 |
| 数量与状态 | 通过 | 成本 / 时效影响由系统根据受影响范围和现场事实推导，不要求业务或现场人员手算差异。 |
| 扫码与识别 | 通过 | 本页为管理端审查对象，不作为扫码执行页；下游 PDA 或现场执行端仅展示关联单据、对象和新旧口径。 |
| 防错 | 通过 | 审核中默认只锁定受影响范围，高风险才整单暂停，避免把局部变更扩大成整单停摆。 |
| UI 样式 | 通过 | 管理端允许较高信息密度，但关键风险、锁定范围、成本 / 时效影响和下游处理动作需保持突出。 |
| 组件交互 | 通过 | 适合使用管理端筛选、分组、抽屉和日志，不要求一线员工在多个技术概念中选择。 |
| 协作关系 | 通过 | 页面把跟单、买手、工艺、仓管、裁剪、印花、染色、车缝、结算、计划履约的影响范围拆开，支持各角色按自己负责的单据处理。 |
| 异常与追溯 | 通过 | 变更意图、影响范围、审核结果、锁定范围、处理动作、成本 / 时效影响和操作记录可追溯。 |
| 现场设备可用性 | 通过 | 下游执行端只看与自己有关的单据处理和新旧口径，不要求理解完整技术包版本模型。 |

## 4. 问题标签

命中标签：状态抽象、点错风险、字段过载、追溯不足。

## 5. 主要问题与处理

| 问题 | 标签 | 影响角色 | 处理方式 | 是否仍有风险 |
| --- | --- | --- | --- | --- |
| 如果要求业务人员先选择「版本关系变更」或「生产补丁」，会把系统概念前置给跟单和现场协作角色。 | 状态抽象 | 跟单、买手、工艺、计划履约 | 页面按变更意图和现场事实进入，系统反推版本关系变更、生产单层补丁以及成本 / 时效影响。 | 否 |
| 如果审核中默认整单暂停，局部变更会不必要地阻断裁剪、印花、染色、车缝或仓储执行。 | 点错风险 | 仓管、裁剪、印花、染色、车缝、计划履约 | 审核中默认只锁定受影响范围，高风险才整单暂停。 | 否 |
| 如果下游执行端展示完整技术包版本模型，一线人员容易读不懂或误处理。 | 字段过载 | 仓管、裁剪、印花、染色、车缝 | 下游执行端只看与自己有关的单据处理和新旧口径，不要求理解完整技术包版本模型。 | 否 |
| 如果成本和时效影响只作为备注保存，结算和计划履约后续无法追溯。 | 追溯不足 | 结算、计划履约、跟单 | 成本 / 时效影响作为系统推导结果进入处理记录，与变更意图、影响范围和审核结论一起追溯。 | 否 |

## 6. 追加审查记录

| 审查项 | 结论 | 说明 |
| --- | --- | --- |
| 场景与系统反推结果一致性 | 通过 | 详细变更单的系统反推结果必须与 80 个场景目录中的预期结果一致，避免业务人员看到同一场景在列表和详情中出现不同处理口径。 |
| 仅成本 / 结算差异覆盖 | 通过 | 仅成本类变更必须生成料工费差异明细，保证结算人员能看到材料、人工或费用差异、责任归因和结算处理方式。 |
| 变更单详情切换 | 通过 | 变更单列表的「查看详情」按变更单 ID 切换当前闭环详情，不再固定展示第一条，方便业务人员逐单查看生产影响、单据处理、成本和时效影响。 |
| 成本与时效明细覆盖 | 通过 | 存在成本差异的变更单必须有料工费差异明细，版本关系或生产补丁执行类变更单必须有时效影响明细，避免列表有结论但详情无账。 |
| 变更单列表分页 | 通过 | 25 条详细变更单可通过上一页 / 下一页翻看，不把后续 mock 数据藏在不可操作的分页后。 |
| 翻页详情一致性 | 通过 | 翻到第 2 页或第 3 页时，闭环详情默认跟随当前页第一条变更单，避免列表已翻页但详情仍停留在上一页造成误判。 |
| 明细唯一性 | 通过 | 生产影响、单据处理、料工费差异和时效影响明细 ID 必须唯一，避免后续定位、导出或追溯时同一 ID 指向不同对象。 |
| 场景覆盖面板移除 | 通过 | 80 个业务场景目录属于规格文档和检查脚本的覆盖范围，不作为管理端演示面板展示，避免把文档说明挤占业务操作页面。 |
| 搜索与统计联动 | 通过 | 搜索栏独立放在统计卡片上方，搜索生产单、变更单、需求单、SPU、款式或负责人后，变更单统计和生产单统计均按命中结果计算。 |
