## 3. 业务与交互结论

- 补料单挂在裁片单下，一张裁片单允许多次补料；每张补料单独立显示“第 N 次”标签，状态仅为“未完成 / 已完成”。
- 操作栏只允许 radio 单选一张未完成补料单，未选中前禁用确认；标签详情也可完成当前这一张。完成一张不连带完成其余补料单，也不改变裁片单主状态。
- 已完成补料单不再展示完成入口；完成前有确认层。重复完成、记录缺失均给出中文反馈；弹层支持 Escape、Tab 圈定和关闭后焦点恢复。行仍存在时回到当前行操作；“有未完成”筛选下末单完成导致行消失时回到“补料完成状态”筛选。
- 新增补料弹窗只选择裁片单；所属生产单作为只读上下文展示，不再提供“按生产单 / 按裁片单”双 Tab，也不保留说明区。
- 裁片放行目标快照只带入生产单、矩阵版本、缺口数据及来源裁片单候选；进入补料管理后仍停在“选择裁片单”，不会用生产单号或 `release-snapshot:*` 直接建单。用户单选一张可用真实裁片单并点“下一步”后，才把快照缺口带入填写页。
- 快照候选按“快照生产单 → 裁片放行记录来源裁片单号 → 当前裁片单候选”求交集。当前 PO14671 快照匹配 CUT14671-A 与 CUT14671-B；A 可选择，B 因已关闭保持禁用。多张必须人工单选；零匹配、全部关闭或快照无效均停留选择页并中文阻断。
- “无补料”只匹配零张补料单，并排除“有未完成 / 全部已完成”等完成类筛选；`ALL_COMPLETED` 定义为至少有一张补料单且每张均已完成，混合状态不属于全部已完成。
- 两个菜单页面读取同一组 15 条固定补料事实；原 12 条补料管理 Mock 与 CUT14671-B 的 3 条记录已集中到共享 fixture 和同一幂等初始化入口。先访问任一页面、重复访问或冷启动，补料总数、关联裁片单、每单序号、状态、原因、明细与物料需求均一致。
- 共享 registry 保存原因说明、完整颜色 / 尺码 / 数量明细和物料 SKU / 名称 / 需求数量；写入时即深拷贝调用方输入，查询、登记和完成返回值也均为嵌套防御性副本，页面既不能通过原始输入引用，也不能通过返回数组污染共享事实。
- 裁片单标签详情完整展示原因、原因说明、补料明细、物料需求、总量及创建 / 完成信息；确认层明确单号、第 N 次、裁片单号和总补料数量。操作栏单选项同时展示单号、次数、创建时间、原因、明细摘要和总量，长明细仅在可滚动弹层展开，不增加列表列宽。
- 历史裁片单缺少可信数量时统一显示“未提供”，不把未知值伪装为真实 0，也不臆造任务边界。
- 裁片单补投影不再特判固定编号：仅当主投影缺失、registry 已存在关联补料、且真实裁片进度候选存在时补入该裁片单。由放行快照新建到 CUT14671-A 的补料可在同一 SPA 的裁片单列表形成标签、打开详情并继续完成；无补料事实或无真实候选的对象不会被臆造。
- 补投影以同一条 registry 事实中的 `cutOrderId + cutOrderNo` 完整身份对严格匹配真实候选；ID 与单号交叉、缺失或分别命中 A / B 时均不降级为 OR 匹配，避免同生产单下裁片单串投影。
- 补料管理的详情、完成确认、列设置和创建确认弹层均按最上层处理初始聚焦、Tab / Shift+Tab 圈定、Escape 逐层关闭和触发点恢复；键盘用户不会落到遮罩后的分页或表格操作。
- “补料对象”只存在裁片单这一种真实归属，因此删除了恒为空的“生产单 / 裁片放行目标快照”筛选项；生产单号和放行快照仍分别作为只读上下文与候选依据，不冒充归属类型。
- 详情确认完成时优先更新当前补料单详情，再于下一帧更新列表、统计与分页；避免低性能运行环境被整表计算阻塞，且列表最终状态、筛选结果与跨菜单共享事实保持一致。

| CUT14671-A 已有补料事实但裁片单主投影缺行，无法展示标签和完成闭环 | `协作断裂`、`追溯不足` | 裁床主管、生产管理人员 | 使用“主投影缺失 + registry 有关联补料 + 真实进度候选存在”的动态补投影规则，不再硬编码 A / B；同生产单按裁片单 ID / 单号匹配真实进度记录 | 否 |

| “生产单 / 裁片放行目标快照”作为补料对象筛选恒为空 | `看不懂` | 生产管理人员 | 删除无结果筛选项；保留生产单关键词与只读上下文、快照候选来源 | 否 |
| 裁片单 ID 与单号来自不同对象时，扁平化 OR 匹配会同时补投影 A / B | `选不对`、`协作断裂` | 裁床主管、生产管理人员 | 传递完整只读身份对，规范化后要求同一 pair 的 ID 与单号同时匹配；缺字段不回退 | 否 |
| 详情完成时先同步刷新整张列表，GitHub 低性能运行环境连续 3 次超过 200ms | `组件误用` | 裁床主管 | 先局部更新当前详情，再于下一帧刷新列表；新增“详情必须先于列表变化”的确定性回归 | 否 |
