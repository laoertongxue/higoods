| 角色匹配 | 通过 | 页面定位为 WLS 管理端 / 主管端看板，面向仓储主管、计划、采购跟单和印花 / 染色仓管，不作为 PDA 一线执行页。 |

| 信息架构与导航 | 通过 | 页面归属 WLS 仓储管理，按「目标面料 -> 库存 / 原料 / 在途 / 异常」组织，符合管理和主管看板的信息查看路径。 |
| 页面模式 | 通过 | 使用看板、筛选、汇总卡片和明细列表承载主管侧判断，不照搬 PDA 扫码执行模式。 |
| 信息负荷 | 通过 | 管理端允许中高密度信息；页面信息围绕 SKU、仓库、数量、缺口、异常和责任人收敛，没有扩展到全仓流水或真实单据创建。 |
| 文案 | 通过 | 文案使用中文业务语义，区分目标面料、直裁面料、印染原料、加工在途和异常原因，避免用英文状态码或抽象系统词直接展示。 |
| 数量与状态 | 通过 | 库存、原料、在途、缺口和差额均按业务单位展示；异常预警展示触发原因、差额和当前状态，避免让用户自行心算。 |
| 扫码与识别 | 有条件通过 | 本看板不是现场扫码页，不要求扫码枪识别；若后续扩展为仓管执行页，需要另做扫码、库位、物料标签和扫错阻断。 |
| 防错 | 有条件通过 | 已通过异常规则、差额、责任人和调拨方向降低误判风险；真实调拨单、采购单创建不在首版范围，后续若开放执行动作必须增加二次确认和操作记录。 |
| UI 样式 | 通过 | 页面按企业后台 / 主管看板风格组织，异常突出但不过度装饰，适合仓储主管和计划人员快速扫描。 |
| 组件交互 | 通过 | 筛选、看板卡片、列表和异常入口符合管理端轻交互要求，没有引入复杂表单或真实工作流。 |
| 协作关系 | 有条件通过 | 页面展示计划、采购、仓储、印花 / 染色仓管之间的缺口和责任人，能支撑跟进协作；真实调拨 / 采购单据流转需后续独立设计。 |
| 异常与追溯 | 有条件通过 | 六类异常展示触发规则、解除条件、差额、责任人和跟进对象，支持主管追溯判断；照片、扫码证据和真实单据处理不在首版范围。 |
| 现场设备可用性 | 通过 | 当前为管理端 / 主管端 Web 看板，主要面向电脑使用；未按 PDA 小屏执行页设计。 |

## 5. 主要问题与处理

| 问题 | 标签 | 影响角色 | 处理方式 | 是否仍有风险 |
| --- | --- | --- | --- | --- |
| 直裁面料和印染原料容易在同一库存视图中混淆，导致计划或仓储主管误判可用量。 | 选不对 | 仓储主管、计划、采购跟单 | 页面明确区分目标面料 SKU、直裁面料库存、印染原料库存和加工在途，不把原料库存直接当成目标面料可用库存。 | 否 |
| 调拨方向如果只展示「需调拨」而不展示来源和目的仓，仓管可能误判从哪里调到哪里。 | 协作断裂 | 仓储主管、印花 / 染色仓管 | 调拨方向已按「中央仓面料仓 -> 对应目的仓」展示，辅助主管判断缺口和仓管跟进范围。 | 否 |
| 页面如果直接创建真实调拨单或采购单，会把管理看板变成执行流，超出首版原型边界。 | 点错风险 | 仓储主管、采购跟单、印花 / 染色仓管 | 首版仅展示缺口、异常、责任人和跟进方向，不创建真实调拨单 / 采购单；后续如扩展执行动作，需另做 PDA / 仓管执行页和确认机制。 | 否 |

| 规则说明有调拨方向但列表异常文案没有直接展示，用户仍需自行推断。 | 协作断裂 | 仓储主管、印花 / 染色仓管 | 已将三类待调拨异常的可见触发文案补充为「中央仓面料仓 -> 对应目的仓」，并用守门脚本检查页面可见。 | 否 |
