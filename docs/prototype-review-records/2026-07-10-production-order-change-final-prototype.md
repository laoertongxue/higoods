## 6. 最终结论

结论：通过

说明：

- 第一阶段已完成统一四步；候选需求明细直接来自当前事实，预选生产单也必须先停留在第一步查看事实。
- 人工只填写需求明细数量或新面料用于生产的件数，不填写生产进度、总数量或面料米数。
- 历史状态和备注中的旧审核口径只在展示层归一，历史单号、影响范围、锁定状态及底层源数据均不改写。
- 新面料候选已排除原面料，同值状态会在内容步骤显示错误并阻断继续。
- 第四步为原型同步模拟：一次函数调用完成锁定、事实核对、处理动作、双向留痕和统一提交或全部回滚，不使用异步机制，不夸大为真实后端事务能力。
- 锁定范围内完成事实写入、最终记录保存和单据留痕索引更新；任一步异常均先恢复执行前事实与记录快照，再形成回滚结果并释放锁。
- 事实重核覆盖当前生产单、纳入范围的后续生产单和第三步处理对象；单据留痕可按单据号反查来源变更单，但各上下游业务页面的统一组件接入仍属于真实系统实施工作。
- 数量变更按每条需求明细分别处理；新增明细的正式版本覆盖关系由系统判断，目标低于已完成数量时只要求跟单确认超出成品去向。
- 全部数量替换必须覆盖需求总数，剩余数量替换不得超过剩余待生产数量；已开工后续生产单选择全部替换时需另行确认旧料和在制品去向。
- 第三步展示当前及后续生产单的真实关联单据状态、计划、完成和待处理数量；最终记录保存全部受影响生产单快照。
- 待执行只说明变更尚未生效，以及确认后会在同一次操作内锁定范围并返回结果，不显示正在变更或当前只读。
- 精确锁冲突提示“生产单正在变更，请稍后再试”由领域与事件检查覆盖；同步浏览器流程验证锁定处理范围、全部成功才生效和五个执行步骤，不制造不可观察的瞬时执行态。
- 成功显示“执行完成，锁定已释放”；回滚显示“已全部回滚，锁定已释放，本次没有生效”，两种终态均不再显示待执行或正在锁定文案。
- 相同处理范围发生嵌套执行时，内层直接返回锁冲突回滚结果，不新增或删除外层持有的锁；外层结束后才释放锁。
- hook 或内部同步执行异常统一转成标准回滚结果；无效输入在分配记录 ID 前阻断，执行失败保存回滚记录并允许同单重试。
- 第四步不显示新增页 header 的“下一步”，待执行和回滚状态分别由“确认执行”和“重新执行”承担主操作。
- 成功结果为“全部处理成功并已统一生效。”，失败结果为“执行失败，本次没有修改任何单据。”；回滚后可在第四步同步重新执行，成功后阻止重复执行。
- 旧记录页统一只读，不显示保存草稿或保存变更内容；页面提供返回详情和“按原记录新建变更”。
- 旧记录按原记录差额展示可匹配数量行，保持当前需求行数和原始属性；无法匹配的旧行不会转成新增需求，并展示具体警告。
- 无法映射的物料文本以“原记录”兼容项保留；兼容前缀不同但标准化文本相同仍会阻止原、新面料同值。
- 原面料候选来自当前生产单面料事实，新面料候选来自系统面料主档；初始和确认件数变化后 state / DOM 使用同一归一数据，手工调整期间如不守恒则如实显示差额并阻断下一步。
- 历史自由备注未改写且不在新表单展示，仅结构化状态做精确展示映射。
- 第三步已按系统建议展示处理方案，但系统建议和 `confirmedMode` 只用于结果预估，不代表跟单已确认；判断初始为空，跟单明确选择后才可继续。
- 跟单选择系统建议时无需填写原因，偏离建议时必须补充现场事实；FULL 旧料去向同样必须显式选择。
- 事件层四步顺序固定为 `order / content / handling / execution`；生产单变更关系页不再暴露独立版本变更或独立补丁入口，统一进入四步流程。
- 高频输入已跳过整页重绘，数量差异、目标合计、物料分配合计和错误提示均采用局部 DOM 更新。
- 按原记录新建会保留生产单、变更类型、原因及可安全映射的变更内容，同时清空旧判断和执行状态。
- 当前事件行为除纯 helper 与 DOM selector 契约外，已由任务 8 Playwright 覆盖四种物料方式与范围组合、必要判断阻断、数量原因传递、失败回滚、同单重试、列表与五模块详情、双视口各阶段溢出和高频输入不整页重绘。
- 已完成人工浏览器复核：列表场景文案无“适用批次/适用颜色尺码”，预选生产单仍从第一步开始，第二步数量明细与第一步当前事实一致，完成记录可只读查看完整事实，技术包关联页不再保留绕过四步流程的旧入口。
- 最终主列表固定九列并分页，稳定种子覆盖数量变更已完成、替换物料待确认执行和已回滚三类记录。
- 主入口两张场景卡明确展示“选择生产单 → 填写变更内容 → 确认处理方案 → 同步执行”，与新增页统一四步一致。
- 最终详情只读取工作流记录 API，固定展示变更内容、当前事实、处理方案、执行结果和相关单据留痕。
- 系统可判断处理项在详情中只读展示；必要事项仅展示跟单决定，不恢复处理人或审批字段。
- 第四步执行成功或回滚后保存为最终记录，表单继续停留在第四步展示原执行结果；记录仓库为原型内存数据并使用深拷贝隔离。
- 同一表单首次执行后持有唯一记录 ID；回滚重试更新同一记录，状态从已回滚覆盖为已完成且列表数量不增加。新表单、换生产单或换场景会清空该 ID。
- 当前事实以完整快照进入最终记录，历史详情不重新读取实时域数据；未执行状态不展示执行后留痕，回滚记录明确说明变更未生效。
- 记录 ID 根据本次发起日期生成并按同日期续号；测试仓库可显式复位或替换，13 条记录已验证第二页数据和详情入口。
- 发起时间与执行时间独立保存；09:00 回滚、10:30 成功重试的同一记录仍保留 09:00 发起时间，最近执行时间及最终单据留痕更新为 10:30。

## 7. 原型边界与上线前置条件

- 当前锁、记录编号预留、事实写回和回滚均为单个浏览器运行时内的同步原型能力，不能替代服务端唯一约束和数据库事务。
- 当前原型不具备跨标签页、跨窗口、跨进程的权威互斥；真实上线必须由服务端对生产单及关联单据加锁，并在同一事务内提交或回滚。
- 当前原型没有真实登录、对象级权限和接口鉴权；直接路由访问不代表生产环境允许访问。
- 当前表单草稿保存在页面内存，刷新后不恢复；同步用户路径只持久化最终成功或回滚结果，不伪造完整五状态持久化。
- 当前成功写回只模拟需求事实更新和物料变更历史追加；真实上下游单据更新必须由后端事务服务完成并返回逐单据结果。
