# AGENTS.md

## 1. 项目定位

这是一个 **产品原型仓库**，不是可交付的业务系统。

目标：
- 快速生成和迭代 HiGood 相关系统页面原型。
- 风格接近 Axure 输出：重 UI、重信息结构、重交互演示。
- 通过 GitHub 自动部署到 Vercel 供在线查看。

默认原则：
- 只做页面结构、轻交互、演示数据。
- 不做真实后端、真实权限、真实状态机、真实数据库设计。
- 不做 API 对接、状态管理体系重构、领域模型重构。
- 不做“为未来可扩展”而提前引入的复杂抽象。

---

## 2. 技术栈与渲染架构

默认技术栈固定为：
- Vite
- TypeScript
- Tailwind CSS
- Vanilla TypeScript 字符串模板渲染

当前项目核心渲染模式是：
- 页面函数返回 HTML 字符串。
- 外壳通过 `renderAppShell(state, pageContent)` 生成。
- 页面最终写入 DOM，例如 `root.innerHTML = ...`。
- React 仅用于少量隔离场景，例如二维码挂载；不要把页面迁移成 React 组件体系。

默认不引入：
- Next.js
- shadcn/ui、Radix UI 或需要 React JSX 组件体系的 UI 框架
- Redux / Zustand / MobX / Provider 架构
- React Query / SWR / 复杂远程数据层
- 后端服务、接口层、鉴权体系
- 复杂表单引擎
- 复杂图表库，除非页面明确需要

说明：
- 这是原型项目，不是工程能力展示项目。
- 优先静态页面 + 本地 Mock 数据 + 轻量交互。
- 所有页面都应以“看起来像真实系统”而不是“实现真实系统”为目标。

---

## 3. 全局工作方式

默认目标是 **在最小必要范围内把当前需求做干净**。

执行判断：
- 如果模块干净、改动小，优先局部 patch。
- 如果模块已经补丁叠补丁、条件分支绕、继续 patch 会更复杂，允许在当前需求直接相关模块内干净重写。
- 重写不是扩大范围；重写必须保持路由、数据格式、事件入口等外部接口兼容，不能顺手改无关模块。

每次任务默认遵循：
1. 先理解当前页面 / 模块现状。
2. 识别本次直接相关的文件和业务链路。
3. 判断采用局部 patch 还是当前模块内重写。
4. 优先复用已有样式、布局、组件和本地 Mock 数据。
5. 不主动重构无关代码。
6. 不主动优化与当前任务无关的目录结构。
7. 不主动升级依赖。
8. 不主动引入新的基础设施。

除非用户明确要求，否则：
- 不新建大型通用组件体系。
- 不抽象“未来可能复用”的工具层。
- 不把页面原型改造成工程化项目。

---

## 4. 输出与改动边界

每次改动必须遵守：
- 优先控制影响范围。
- 优先保留现有可运行页面。
- 不修改无关模块。
- 不把一个小需求扩写成系统级重构。
- 不为兼容混乱旧代码继续堆叠兼容层；该模块已乱时，按第 3 节在当前范围内干净重写。

如果需求较大或模块需要重写，默认分步执行：
- 先做页面结构。
- 再补交互。
- 再补 Mock 数据。
- 最后做收口优化和验证。

## 5. 原型设计原则

### 5.1 页面目标

所有页面必须优先表达：
- 信息结构
- 业务关系
- 状态变化
- 页面跳转
- 关键动作入口
- 演示所需的 Mock 数据

### 5.2 不做的东西

默认不做：
- 真实业务校验已完成。
- 复杂接口联调。
- 复杂状态同步。
- 复杂缓存。
- 复杂错误恢复机制。
- 复杂上传系统。
- 复杂图像处理。
- 复杂权限控制。

### 5.3 Mock 数据原则

Mock 数据必须：
- 足够支撑页面阅读和交互。
- 能体现上一步与下一步关系。
- 能体现不同状态。
- 能体现异常 / 风险 / 边界场景。
- 不要只有空壳页面。
- 不要只有一两条极少数据。

Mock 数据应优先表现：
- 中文业务语义。
- 真实字段组合。
- 状态差异。
- 典型异常。

### 5.4 印尼工厂现场协同设计治理

任何涉及页面原型、交互、文案、UI 样式、组件使用、Mock 数据、路由入口、现场执行链路的更新，都必须先参照以下两份项目级文档：

- `docs/higood-indonesia-factory-product-design-guidelines.md`
- `docs/higood-indonesia-factory-prototype-review-checklist.md`

执行要求：

1. 设计前必须按《HiGood 印尼工厂现场协同系统产品设计规范》确认角色、端类型、现场能力假设、协作关系、页面模式、文案、UI 样式、防错和异常兜底。
2. 完成后必须按《HiGood 印尼工厂现场协同系统原型审查清单》做人工自查。
3. 只要改动涉及 `src/pages/`、`src/components/`、`src/data/`、`src/router/`、`src/main-handlers/` 中的原型页面、组件、Mock 数据、路由或交互处理，就必须新增或更新一份审查记录，模板见 `docs/prototype-review-record-template.md`，记录放在 `docs/prototype-review-records/`。
4. 最终回复必须说明本次改动参考了哪几类规范、自查结论，以及是否存在例外。
5. 提交或交付前必须运行 `npm run check:prototype-design-governance`。如需检查整个工作区，可运行 `npm run check:prototype-design-governance -- --all`。

不得用“只是小改”“只是文案”“只是 Mock 数据”绕过上述要求。印尼工厂现场端的文案、字段、按钮、状态、颜色、数量和异常提示都属于产品设计的一部分。

## 6. 中文化要求

这是中文业务原型项目，必须严格遵守：
- 页面文案全部中文展示。
- 状态值全部中文展示。
- 禁止英文状态码直接出现在页面上。
- 禁止出现乱码。
- 时间、金额、单位、状态统一用中文业务语义表达。

例如：
- 不显示 `PENDING`, `DONE`, `IN_PROGRESS`。
- 要显示 `待处理`、`已完成`、`进行中`。

## 7. UI 与交互要求

默认遵循以下原则：
- 风格简洁、清晰、偏企业后台。
- 优先可读性，不追求炫技。
- 页面布局稳定，不频繁变换交互范式。
- 保持模块内风格统一。
- 轻交互优先：抽屉、弹窗、页签、筛选、详情展开。
- 不做复杂动画。
- 不做重视觉营销风格。

默认要关注：
- 列表是否可读。
- 卡片是否能承载关键信息。
- 筛选是否足够用。
- 详情是否能承接列表入口。
- 页面之间是否能连起来。

### 7.1 组件使用规则

项目已有字符串模板 UI 组件库，位于 `src/components/ui/`。新增或重写页面时优先使用现有组件：
- `button.ts`
- `badge.ts`
- `dialog.ts`
- `drawer.ts`
- `table.ts`
- `form.ts`
- `pagination.ts`
- `filter-bar.ts`
- `card.ts`
- `toast.ts`
- `tabs.ts`
- `steps.ts`
- `dropdown.ts`
- `collapse.ts`

规则：
- 不引入 shadcn/ui；它依赖 React JSX 组件体系，不符合当前渲染架构。
- 不为了 UI 统一把页面迁移到 React。
- 按钮、表格、弹窗、抽屉、表单、筛选、分页、状态徽章等常见 UI，优先复用 `src/components/ui/`。
- 如果现有组件不能覆盖，必须先判断是否应沉淀为 `src/components/ui/` 组件：
  - 按钮组合、表格操作、弹窗、抽屉、筛选、Tabs、Steps、Dropdown、Toast、Collapse、空状态、加载态、确认提示等通用 UI 模式，应先补充或扩展组件库，再在页面中调用。
  - 强业务、只在当前页面出现一次、字段结构与当前业务数据深度耦合的 UI，可以在当前页面内做局部 `renderXxx()` 实现。
  - 页面内局部实现也必须拆成清晰函数，禁止直接堆进大段页面模板。
- 避免在大型页面里继续堆大量内联 `<button class="...">`、`<select class="...">`、表格和弹窗模板。

### 7.2 渲染性能与交互响应规则

当前项目采用 Vanilla TypeScript + 字符串模板渲染。页面迭代时必须控制重渲染范围，避免把轻量交互做成整页重绘。

默认规则：
- 任一页面的任一功能按钮，点击后响应速度不得高于 200ms；只有满足这一规则，才算完成相应的页面功能开发。
- 页面级交互优先采用局部重渲染或局部 DOM 更新；只有路由切换、页面主体数据整体变化时才允许整页重渲染。
- 弹窗、抽屉、Dropdown、行操作、Tab 切换、展开 / 收起等轻交互，尽量不要触发 `root.innerHTML` 级别的整页重绘。
- 任何数据列表、数据宽表、明细表、日志列表都必须有分页；即使当前只是 Mock 数据、首版数据量很少，也必须保留分页控件和页码 / 每页条数口径。只有纯展示型小卡片组、固定数量摘要区、明确不是“列表”的静态说明区可以不分页。
- 表格、列表、日志、明细行较多时，还必须限制首屏渲染量，必要时采用懒渲染 / 虚拟化思路；禁止一次性把大量不可见数据全部铺到 DOM。
- 高频输入场景，例如搜索框、数量输入、备注输入、筛选输入，应使用跳过重渲染、局部更新或 debounce，避免每次输入都触发完整页面重绘。
- 输入事件在每次 `input` 后不允许触发整页重渲染；必须使用局部 DOM 更新、跳过重渲染或 debounce 等方式处理。
- Icon hydration 只应作用于新插入的 DOM 区域；局部更新时不要重新扫描整页图标。
- 大型页面必须拆分渲染函数，避免单文件中堆积巨型 HTML 模板；稳定不变的静态片段可以缓存或复用，减少重复字符串构造。
- 不为了性能引入 React、状态管理框架、虚拟 DOM 或复杂基础设施；优先在现有字符串模板架构内做局部优化。

### 7.3.1 列表页变更硬门禁

标准列表页模板的要求必须通过自动检查执行，不能只依赖人工约定：

- 新增或调整的列表页必须在页面源文件顶部声明 `// @page-pattern: list`，并通过 `npm run check:list-page-governance`。
- 列表页必须使用标准列表页组件契约：`renderStandardListPage`、`renderStandardListTable` 和 `renderTablePagination`，同时具备分页、列排序 / 显示 / 冻结能力与右侧固定操作栏。
- `scripts/standard-list-page-baseline.json` 只允许保留尚未迁移且未发生变化的历史页面哈希。基线哈希不得被修改，也不得为绕过检查新增；完成迁移后应删除对应基线条目。
- `npm run build` 会自动先执行统一列表页治理检查；Pull Request 和推送到 `main` 必须通过 GitHub Actions 的 `list-page-governance` 必检状态。
- 例外只能通过原型审查记录说明范围、原因和迁移计划，不能通过修改检查脚本或基线文件绕过门禁。

### 7.3 标准列表页模板治理

标准列表页模板以补料管理页面 `/fcs/craft/cutting/supplement-management` 为验收基准。核心公共组件为 `src/components/ui/list-page.ts`，表格渲染与列偏好模型分别位于 `src/components/ui/list-table.ts`、`src/components/ui/list-table-model.ts`。

强制规则：

- 后续所有新增列表页，以及被调整的既有列表页，都必须以该模板的样式、公共组件和交互为基准；调整既有列表页的结构、筛选、统计、表格或分页时同步向模板对齐。本任务不批量迁移其他既有页面。
- 必须优先复用公共组件，不得复制页面模板；公共骨架或表格能力不足时，先在 `src/components/ui/` 内补充最小通用能力，再由业务页面组合使用。
- 标准列表摘要卡片必须采用 48px 单行布局，标签和值水平排列，不得使用上下两段造成额外垂直占用。
- 所有可排序列必须显示未排序、升序和降序三态图标；图标必须由组件直接输出，局部刷新后仍保持可见。
- 所有数据列表必须分页；即使当前只有少量 Mock 数据，也必须展示分页控件，并明确当前页、每页条数和总数口径。
- 当列总宽超过表格可视区、需要横向滚动时，必须支持显示列选择、列顺序拖拽调整、普通冻结列固定左侧和数据排序。
- 用户冻结的普通列必须立即进入表格最左侧固定区，从横向滚动开始到结束都不移动；多列冻结按用户列顺序排列，取消冻结后恢复普通列位置。
- 必需列和业务防错列必须声明为不可隐藏的“必需列”（代码字段为 `required`）；操作列也必须保持可见。
- 操作列必须固定在右侧且不随横向滚动，并始终保持可见和可操作。
- 列显示、列顺序、冻结列和每页条数必须按路由持久化；当前页和数据排序不得持久化，刷新或重新进入页面时回到稳定默认状态。
- 任一模块不得出现无业务逻辑的说明性文案；允许展示真实业务状态、风险、异常和空态，以及完成当前任务所必需的操作反馈。
- 以 1366×768 为标准验收分辨率，1280×720 为最低可用分辨率；页面主体不得产生横向溢出，宽表必须在表格容器内部滚动。
- 任何无法遵循上述规则的业务例外，都必须在对应的 prototype review record 写明理由、影响范围和替代防错措施。

## 9. 模块实现约定

### 9.1 FCS / HiGood 页面

这些页面默认是产品原型，不是生产系统：
- 侧重页面表达。
- 侧重链路展示。
- 侧重信息和状态。
- 侧重跨页面跳转。
- 侧重演示价值。

### 9.2 字典类页面

字典页默认按 **结果展示页** 处理，不做重后台维护感。

除非用户明确要求，否则：
- 不默认开放新增。
- 不默认开放批量操作。
- 不默认开放复杂编辑能力。
- 优先展示结果、来源、引用关系。

### 9.3 技术包 / 任务 / PDA 页面

默认：
- 重点是页面可读性和业务已完成演示。
- 用 mock 数据体现来源、状态、上一步与下一步。
- 不引入复杂底层逻辑。

### 9.4 PDA 页面执行优先标准

PDA 页面是现场执行工具，不是管理后台。所有 PDA 页面默认面向低培训成本的一线操作员，尤其是印尼本土工厂现场人员，必须以“马上知道做什么、对什么做、点哪里做”为首要目标。

默认原则：
- 业务模型必须保持正确，不能为了页面简单而扭曲真实对象关系。
- PDA 表达必须尽量傻瓜化：少层级、少字段、少解释、少抽象词。
- 页面核心必须动作优先：主按钮清晰、入口靠前、当前步骤明确。
- 首屏只展示执行必需信息，例如当前任务、当前对象、关键物料 / 数量 / 状态、下一步动作。
- 管理视角信息、链路追踪、复杂来源、历史日志、统计分析等内容默认收起、后置或放到详情入口。
- 文案必须短、直接、动作化，避免 `执行对象`、`投影`、`链路`、`来源记录` 等抽象表达直接出现在 PDA 页面。
- 当业务结构复杂时，PDA 可以用简化展示，但必须保留防错所需的关键归属信息。例如裁片链路可保持 `裁片任务 -> 裁片单 -> 铺布单` 的正确模型，但现场页面应突出可操作的铺布单和主动作入口。
- 如果只有一个业务对象，不要为了表达完整层级而制造多余分组；如果有多个对象，分组只服务于避免选错和快速进入操作。

## 10. 任务执行默认格式

当用户要求改页面时，默认按以下方式执行：

1. 先识别本次只涉及哪些页面 / 文件。
2. 按第 3 节判断：局部 patch，或当前模块内干净重写。
3. 补足必要 Mock 数据。
4. 保持页面可演示。
5. 不扩大到无关模块。
6. 最终给出：
   - 改了哪些文件。
   - 做了什么，采用 patch 还是重写。
   - 如何验证。

## 11. 验证原则

每次改动后，尽量验证：
- 页面是否能正常运行。
- 路由是否可达。
- 交互是否可点。
- 列表与详情是否连通。
- 不出现明显空白页、死链、报错页。
- 中文文案和状态是否正确。

如果项目已有可用命令，则优先运行最小必要验证。

如仓库命令已存在，请使用：
- 开发启动：`npm run dev`
- 构建检查：`npm run build`

如果实际命令不同，请以仓库现有脚本为准，不主动新造命令。

### 11.1 交互性能验证

涉及弹窗、抽屉、Dropdown、表格行操作、筛选、Tab 切换、输入框等交互改动时，除页面可达外，还需要检查：
- 操作后是否发生不必要的整页闪烁或滚动位置丢失。
- 输入框输入时是否明显卡顿。
- 弹窗、抽屉、Dropdown 打开关闭是否只更新必要区域。
- 大表格页面首屏是否加载过重，是否存在一次性渲染大量隐藏内容的问题。
- 图标 hydration 是否限定在新插入区域，避免重复扫描整页。

### 11.2 本地运行查看效果

当用户要求“本地运行查看效果”时，默认含义是：必须让同一局域网内的设备可以访问当前项目，而不是只在本机 `127.0.0.1` 可访问。

执行要求：
- 启动 Vite 时使用：`npm run dev -- --host 0.0.0.0 --port <port>`。
- 优先复用当前演示端口；如果端口被占用，选择其他可用端口并明确告知。
- 获取本机局域网 IP，例如 macOS 可用 `ipconfig getifaddr en0`，必要时再查 `en1`。
- 最终回复必须给出局域网访问地址：`http://<局域网IP>:<port>/<path>`。
- 必须验证本机访问该局域网地址可达，例如 `curl -I http://<局域网IP>:<port>/<path>` 返回 `200`。
- 如局域网设备仍无法访问，提醒检查是否在同一 Wi-Fi / 局域网，以及 macOS 防火墙是否允许 Node / Vite 入站连接。

## 12. CodeGraph 使用规则

本项目已接入 CodeGraph。执行代码核查、方案设计、代码修改、实现确认前，必须优先使用 CodeGraph 辅助定位上下文。

默认流程：

1. 开始前先运行：
   - `codegraph sync`
   - `codegraph status`
2. 核查或修改模块前，根据任务使用：
   - `codegraph files`
   - `codegraph context`
   - `codegraph search`
   - `codegraph explore`
   - `codegraph callers`
   - `codegraph callees`
   - `codegraph impact`
3. CodeGraph 用于确认模块边界、符号关系、调用关系和影响面；中文文案、按钮、路由、状态展示仍需结合 `rg` 精查。
4. 完成代码修改后必须再次运行：
   - `codegraph sync`
   - `codegraph status`
5. 最终回复中说明是否已同步 CodeGraph 索引。

## 13. 明确禁止事项

除非用户明确要求，否则禁止：
- 把原型项目改造成工程化架构样板。
- 大规模重构目录。
- 引入 store / provider / API 层。
- 引入真实服务端依赖。
- 引入 shadcn/ui 或把页面迁移到 React 组件体系。
- 为“以后可能用到”提前做复杂封装。
- 用英文状态码直接出页面。
- 用大段无意义静态假数据灌满页面。
- 为兼容混乱旧代码而堆叠 patch、引入兼容层。

## 14. 与用户协作的默认原则

默认认为用户要的是：
- 快速、准确、低成本地迭代原型。
- 代码清洁、维护简单，模块该重写就重写，不叠补丁。
- 结果可直接部署到 Vercel 演示。
- 页面与业务一致，而不是代码“看起来高级”。

因此：
- 优先满足原型准确性。
- 优先满足业务表达。
- 优先控制改动范围。
- 优先控制复杂度。
